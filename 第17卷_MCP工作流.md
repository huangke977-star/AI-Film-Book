# 第 17 卷：MCP 工作流

## 学习目标

学完本卷后，你应该能做到：

- 理解 MCP 为什么能让 Agent 连接外部工具、文件、浏览器和平台能力。
- 区分 Tool、Resource、Prompt 的基本作用。
- 知道 AI 视频项目里哪些能力适合通过 MCP 或类似工具接入。
- 能设计一个 Codex + MCP 的视频项目工作流。
- 会写 Agent 权限表、工具调用审计表和安全边界检查表。
- 知道密钥、账号、客户素材、浏览器和平台 API 都需要最小权限和人工确认。

本卷不把你变成 MCP 协议开发者。它让你知道：当 Agent 能调用工具时，视频制作流程会怎样变化，以及风险边界在哪里。

## 本卷导读：Agent 需要工具，但工具需要边界

前面我们一直在用 Codex 思路做项目：

```text
读取文件
-> 理解上下文
-> 写脚本
-> 写分镜
-> 生成任务卡
-> 记录复盘
```

如果 Agent 只能读写本地文件，它已经很有用。但真实视频项目还会涉及：

- 浏览器页面。
- 表格和数据库。
- 素材库。
- 生成平台。
- 下载结果。
- 通知系统。
- 自动化工具。
- 客户反馈。

MCP 的价值，是让 Agent 以更标准的方式连接这些外部能力。

但能力越强，边界越重要。

```text
能连接工具
不等于可以随便调用工具
```

## 1. MCP 基本概念

MCP 是 Model Context Protocol，中文可以理解为模型上下文协议。它提供一种方式，让模型或 Agent 连接外部工具和上下文。

在视频项目里，你可以先把它理解成：

```text
Agent 和外部工具之间的插座规范
```

Agent 不需要每次都临时猜怎么连接工具，而是通过 MCP Server 暴露的能力来使用工具。

### 1.1 三个角色

| 角色 | 说明 |
| --- | --- |
| Host | 承载 Agent 的应用，例如 Codex 或其他客户端 |
| Client | Host 里连接 MCP Server 的组件 |
| Server | 提供工具、资源或提示的外部服务 |

新手不必先记实现细节，只要理解：

```text
MCP Server 把能力暴露给 Agent。
Agent 通过 Host 使用这些能力。
```

## 2. Agent 为什么需要工具

纯聊天模型只能基于输入回答。视频项目需要真实操作：

- 读取项目文件。
- 搜索素材。
- 打开网页。
- 调用生成平台。
- 查询任务状态。
- 下载文件。
- 写入复盘表。
- 通知你审核。

这些都不是单纯“写一段话”能完成的。

工具让 Agent 从“建议者”变成“协作者”。

但协作者也要遵守规则：

- 只能访问必要文件。
- 只能调用必要接口。
- 关键动作要确认。
- 操作要留记录。
- 出错要能回滚。

## 3. Tool、Resource、Prompt 的区别

MCP 官方规范中常见三类能力：Tools、Resources、Prompts。

### 3.1 Tool

Tool 是可以执行动作的能力。

例子：

- 创建文件。
- 查询任务状态。
- 下载生成结果。
- 发送通知。
- 调用视频生成接口。

Tool 有副作用，所以要谨慎。

### 3.2 Resource

Resource 是可以读取的上下文。

例子：

- 项目 Brief。
- 分镜表。
- 素材库记录。
- 客户反馈。
- 生成结果列表。

Resource 更像“给 Agent 看资料”。

### 3.3 Prompt

Prompt 是可复用提示模板。

例子：

- 广告 Brief 生成模板。
- 分镜拆解模板。
- 视频生成任务卡模板。
- 失败复盘模板。

Prompt 让团队用同一套表达方式工作。

### 3.4 对比表

| 类型 | 作用 | 风险 |
| --- | --- | --- |
| Tool | 执行动作 | 可能改变文件、消耗费用、触发外部操作 |
| Resource | 提供上下文 | 可能包含客户资料或敏感信息 |
| Prompt | 标准化表达 | 可能把错误模板反复使用 |

## 4. 文件系统 MCP

文件系统是 AI 视频项目最基础的上下文。

适合暴露给 Agent 的目录：

```text
brief.md
script.md
storyboard.md
style_guide.md
character_bible.md
prompts/
outputs/selected/
iteration_log.md
```

不建议随意暴露：

- 密钥文件。
- 客户合同。
- 私人聊天记录。
- 未授权素材。
- 操作系统敏感目录。

### 4.1 文件权限表

```markdown
| 路径 | 权限 | 用途 |
| --- | --- | --- |
| project/brief.md | 读写 | 更新项目目标 |
| project/storyboard.md | 读写 | 生成分镜 |
| project/outputs/raw/ | 只读 | 查看生成结果 |
| project/delivery/ | 人工确认后写入 | 最终交付 |
| project/secrets/ | 不暴露 | 密钥和账号信息 |
```

文件权限不是越大越好。够用就好。

## 5. 浏览器 MCP

浏览器能力可以帮助 Agent：

- 打开工具页面。
- 查看生成进度。
- 读取公开页面信息。
- 做页面截图。
- 辅助核验网页内容。

但浏览器也可能连接你的登录状态，所以风险更高。

### 5.1 使用原则

- 只打开任务需要的网站。
- 不让 Agent 随意操作账号设置。
- 涉及支付、发布、删除、提交的动作要人工确认。
- 不把私人账号页面截图写进公开文档。
- 关键页面信息要记录来源和时间。

### 5.2 视频项目用途

| 用途 | 例子 |
| --- | --- |
| 案例核验 | 打开公开视频页面，记录标题、作者、时长 |
| 工具确认 | 查看官方文档是否说明某功能 |
| 素材下载 | 下载已授权生成结果 |
| 页面截图 | 保存项目证据或客户确认 |

## 6. 平台 API 和生成工具

视频生成平台如果提供 API 或工具接口，MCP 或类似连接方式可以让 Agent：

- 创建生成任务。
- 查询任务状态。
- 下载结果。
- 记录成本。
- 归档文件。

### 6.1 任务卡到工具调用

```markdown
# S01 Task

prompt: 夜晚工作室，创作者看向屏幕，真实纪录片风格。
ratio: 9:16
duration: 4s
reference: REF_workspace_v01.png
```

Agent 可以把任务卡转成工具调用。但在商业项目里，发起高成本生成前最好设置确认点。

### 6.2 结果回写

工具返回结果后，Agent 应该写入：

```markdown
| shot_id | task_id | status | file | cost_note | review |
| --- | --- | --- | --- | --- | --- |
| S01 | task_001 | complete | outputs/raw/S01_v01.mp4 | 待填 | 待人工评审 |
```

这样生成不是黑箱。

## 7. LibTV Skill 或类似工作流

本项目从 Ross 案例出发，强调 Codex 和 LibTV 或同类工具的工作流。无论具体工具叫什么，本质都是：

```text
Codex 负责项目上下文
生成工具负责素材输出
人负责判断和确认
```

### 7.1 工作流

```text
Codex 读取 brief、style、storyboard
-> 生成每个镜头任务卡
-> 调用 LibTV Skill 或类似接口
-> 查询生成状态
-> 下载结果
-> 写入 iteration_log
-> 人工选择 selected/rejected
```

### 7.2 不要跳过复盘

如果 Agent 只是不断调用生成工具，成本会失控。每轮生成后必须写：

- 为什么可用。
- 为什么失败。
- 下一轮改什么。
- 是否值得继续。

## 8. Codex + MCP 视频项目示例

一个可控的视频项目可以这样设计：

```text
project/
├── brief.md
├── script.md
├── storyboard.md
├── style_guide.md
├── character_bible.md
├── prompts/
├── outputs/
├── audio/
├── edit/
└── iteration_log.md
```

MCP 能力：

| 能力 | 类型 | 权限 |
| --- | --- | --- |
| 读取项目文件 | Resource | 只读或指定读写 |
| 写入任务卡 | Tool | 仅 prompts/ |
| 查询生成状态 | Tool | 只查询 |
| 下载结果 | Tool | 写入 outputs/raw/ |
| 读取网页文档 | Tool | 指定域名 |
| 发送审核提醒 | Tool | 指定频道 |

### 8.1 人工确认点

```text
Brief 确认
脚本确认
分镜确认
高成本生成确认
客户交付确认
发布确认
```

MCP 让工具更好用，但不应该拿走这些确认点。

## 9. 权限最小化

权限最小化的意思是：只给任务需要的权限。

### 9.1 示例

如果 Agent 只需要读取素材，不要给它删除素材的能力。

如果 Agent 只需要创建草稿，不要给它发布成片的能力。

如果 Agent 只需要查询生成状态，不要给它修改账号设置的能力。

### 9.2 权限表

```markdown
| 工具 | 允许 | 不允许 |
| --- | --- | --- |
| 文件工具 | 读写项目目录 | 访问密钥目录 |
| 浏览器工具 | 打开官方文档和指定项目页 | 操作支付和账号安全页 |
| 生成工具 | 创建低成本测试任务 | 未确认的大批量生成 |
| 通知工具 | 发送审核提醒 | 向外部客户自动承诺结果 |
```

## 10. 密钥和账号安全

AI 视频工作流可能涉及多个账号：

- OpenAI 或其他 AI API。
- 视频生成平台。
- 音乐平台。
- 素材库。
- 自动化工具。
- 云盘或表格。

基本原则：

- 密钥不写入项目文档。
- 密钥不提交到 Git。
- 客户账号和个人账号分开。
- 测试环境和真实项目分开。
- 重要操作前人工确认。

### 10.1 凭证登记表

```markdown
| 服务 | 用途 | 保存位置 | 负责人 | 备注 |
| --- | --- | --- | --- | --- |
| AI API | 脚本和任务卡 | 工具凭证系统 | 我 | 不写入 Markdown |
| 视频生成平台 | 生成素材 | 平台账号 | 我 | 高成本任务需确认 |
| 云盘 | 交付文件 | 客户授权目录 | 我 | 不放密钥 |
```

这张表不记录密钥本身，只记录管理责任。

## 11. 日志、审计和回滚

工具调用要可追踪。

### 11.1 调用审计表

```markdown
| 时间 | 工具 | 输入摘要 | 输出 | 操作人 | 备注 |
| --- | --- | --- | --- | --- | --- |
| 10:20 | generate_video | S01 task | task_001 | Codex | 待查询 |
| 10:35 | download_result | task_001 | S01_v01.mp4 | Codex | raw |
| 10:40 | write_log | S01_v01 | iteration_log | Codex | 人工待评 |
```

### 11.2 回滚

需要能回答：

- 哪个文件被改了。
- 哪个任务被发起了。
- 哪个结果被下载了。
- 哪个版本是最终版。

Git、版本命名和日志是回滚的基础。

## 12. MCP 能力清单

在视频项目里，可以先做一张能力清单。

```markdown
| 能力 | 类型 | 是否需要 | 风险 |
| --- | --- | --- | --- |
| 读取 Brief | Resource | 是 | 低 |
| 写入脚本 | Tool | 是 | 中 |
| 调用视频生成 | Tool | 可选 | 成本风险 |
| 读取浏览器页面 | Tool | 可选 | 隐私风险 |
| 发布视频 | Tool | 暂不需要 | 高 |
```

新手阶段，把高风险能力先关掉。

## 13. Prompt 作为团队资产

很多人以为 MCP 只和工具调用有关，其实 Prompt 也很重要。

在视频项目里，你可以把常用提示做成可复用资产：

```text
广告 Brief 生成
短剧人设生成
分镜拆解
视频生成任务卡
失败样本复盘
客户修改归类
商业交付检查
```

### 13.1 Prompt 模板示例

```markdown
# Prompt: shot_task_from_storyboard

## 输入

- 项目 Brief
- 风格手册
- 分镜行
- 人物设定

## 输出

- 镜头目的
- 主体
- 场景
- 动作
- 运镜
- 时长
- 禁止项
- 评审标准

## 约束

不要生成平台按钮路径。
不要写成本承诺。
不要让模型生成复杂中文文字。
```

### 13.2 为什么要模板化

模板化不是为了让所有作品一样，而是为了让团队每次都不漏关键字段。AI 视频最怕“临时想起一个要求”。Prompt 模板能把经验固化下来。

## 14. 浏览器核验案例

第 1 卷到第 13 卷一直强调资料来源。浏览器工具可以帮助 Agent 核验网页信息。

以公开视频案例为例，Agent 可以做：

```text
打开视频页面
-> 记录标题、作者、发布时间、时长
-> 观察关键画面
-> 区分页面事实和制作推断
-> 写入资料来源与核验记录
```

### 14.1 核验记录模板

```markdown
# Web Verification Record

## 页面

URL：
核验时间：

## 可直接观察

标题：
作者：
发布时间：
时长：
页面章节：

## 画面观察

关键画面：
字幕风格：
人物和场景：

## 推断

可能使用的工作流：
推断依据：
不确定项：
```

这能防止把“我猜的”写成“真实发生的”。

## 15. MCP 与 n8n 的分工

n8n 和 MCP 不冲突。

可以这样理解：

| 能力 | 更像 n8n | 更像 MCP |
| --- | --- | --- |
| 定时批处理 | 是 | 不一定 |
| 多服务流程编排 | 是 | 可配合 |
| 让 Agent 临时调用工具 | 可配合 | 是 |
| 给 Agent 提供资源 | 可配合 | 是 |
| 复杂人工审核流 | 是 | 可配合 |
| 项目上下文理解 | 不擅长 | 更适合 Agent |

### 15.1 配合方式

```text
n8n 负责定时和流程编排
MCP 负责让 Agent 安全使用工具和资源
Codex 负责理解项目文件、拆解任务、写复盘
人负责关键判断
```

例如：

```text
n8n 每周收集选题
-> 通知 Codex 项目目录
-> Codex 通过 MCP 读取文件和素材记录
-> Codex 生成 Brief 和分镜
-> n8n 把待审核任务推送给你
```

这比让一个工具包办所有事情更稳。

## 16. 客户项目安全评审

当 MCP 工作流进入客户项目时，要做安全评审。

```markdown
| 检查项 | 问题 |
| --- | --- |
| 客户资料 | 是否只暴露项目需要的文件 |
| 素材授权 | Agent 是否能区分授权素材和未确认素材 |
| 账号权限 | 是否使用项目专用账号或最小权限 |
| 工具动作 | 哪些动作会产生费用 |
| 写入范围 | Agent 能写入哪些目录 |
| 删除能力 | 是否关闭或限制删除 |
| 发布能力 | 是否需要人工确认 |
| 审计日志 | 每次调用是否记录 |
```

### 16.1 高风险动作

这些动作要格外谨慎：

- 删除文件。
- 覆盖最终交付文件。
- 批量发起生成任务。
- 操作客户账号。
- 向外部发送客户资料。
- 提交发布内容。
- 改动合同、报价或授权记录。

新手阶段可以把这些动作先排除在工具能力之外。

## 17. MCP 落地检查表

在真正启用 MCP 或类似工具连接前，先填这张表：

```markdown
| 检查项 | 是否完成 |
| --- | --- |
| 项目目录结构清楚 |
| Brief、脚本、分镜已落地为文件 |
| 需要暴露的 Resource 已列出 |
| 需要开放的 Tool 已列出 |
| 高风险 Tool 已关闭或加确认 |
| 凭证保存位置清楚 |
| 审计表已建立 |
| 输出目录已限制 |
| 失败处理方式已写 |
| 人工确认点已写 |
```

这张表比“先接上再说”更重要。

## 18. 最小 MCP 视频工作流

新手可以从最小版本开始：

```text
Resource：读取 brief.md、storyboard.md、style_guide.md
Prompt：分镜转任务卡模板
Tool：写入 prompts/ 目录
Tool：写入 iteration_log.md
```

这个最小工作流不直接调用生成平台，也不下载文件。它先解决一个问题：

```text
让 Agent 根据项目上下文稳定生成任务卡和复盘记录。
```

等这个稳定后，再接入：

- 查询生成状态。
- 下载结果。
- 发送审核提醒。
- 读取浏览器页面。

不要第一天就连接所有工具。

## 19. 团队角色分工

当工具连接变多时，要明确谁负责什么。

```markdown
| 角色 | 负责 |
| --- | --- |
| 创作者 | 选题、脚本判断、最终审美 |
| Codex / Agent | 读取上下文、拆任务、写文档、辅助复盘 |
| 自动化工作流 | 定时、通知、表格记录、状态流转 |
| 生成平台 | 输出图像、视频、音频素材 |
| 客户或审核人 | 确认需求、品牌、产品和交付结果 |
```

不要把所有责任都推给 Agent。Agent 可以执行和辅助，但客户需求、商业判断、发布决定和授权确认必须有人负责。

### 19.1 RACI 简化表

```markdown
| 任务 | 负责人 | 协助 | 确认 |
| --- | --- | --- | --- |
| 写 Brief | 创作者 | Agent | 客户 |
| 拆分镜 | 创作者 | Agent | 创作者 |
| 发起生成 | 创作者 | Agent / Tool | 创作者 |
| 选择素材 | 创作者 | Agent 提供评审表 | 创作者 |
| 商业交付 | 创作者 | 自动化生成清单 | 客户 |
```

只要责任不清，工具越多越乱。

## 20. 常见失败模式

### 20.1 上下文太少

Agent 只看到一句任务，没有 Brief、风格和分镜，输出就会飘。

修正：先把项目文件补齐，再调用工具。

### 20.2 工具太多

一口气连接文件、浏览器、生成平台、云盘、表格、发布平台，排错会很困难。

修正：先接一个 Resource，再接一个低风险 Tool。

### 20.3 没有确认点

Agent 连续做了很多动作，你最后才发现方向错了。

修正：在 Brief、脚本、分镜、高成本生成和交付前设置确认。

### 20.4 没有命名规范

工具下载了一堆默认文件名，后期找不到来源。

修正：输出路径和文件名写进任务卡。

### 20.5 没有复盘

工具调用成功了，但不知道结果为什么好或坏。

修正：每轮生成后写 iteration_log。

### 20.6 上线前最后检查

```markdown
| 问题 | 是否确认 |
| --- | --- |
| 这个工具是否真的必要 |
| 是否限制了目录和账号范围 |
| 是否有人工确认点 |
| 是否记录每次调用 |
| 是否能停止或回滚 |
| 是否不会把客户资料写到公开位置 |
```

如果这些问题答不上来，就先不要把工具放进真实项目。先在测试项目里跑通，再进入客户项目。

MCP 的价值在于可控连接，不在于把所有门一次打开。

少一点权限，多一点记录。

项目就会更稳。

也更容易复盘。

稳。

先稳住。

## 21. 常见错误

### 错误 1：把 MCP 当魔法

修正：MCP 只是连接方式，项目结构仍然要清楚。

### 错误 2：权限给太大

修正：按任务给最小能力。

### 错误 3：没有日志

修正：每次工具调用写入审计表。

### 错误 4：工具输出直接进入成片

修正：所有生成结果先进入 raw，再人工选择。

### 错误 5：把密钥写进文档

修正：使用凭证系统或环境变量，不写入仓库。

## 22. 故障排查

### Agent 看不到文件

处理：

- 检查文件是否在允许目录。
- 检查路径是否正确。
- 检查 Resource 是否暴露。
- 不要让 Agent 访问整个磁盘。

### 工具调用失败

处理：

- 看错误信息。
- 检查输入字段。
- 检查凭证是否有效。
- 用最小样本测试。
- 记录到审计表。

### 输出写错位置

处理：

- 固定项目目录。
- 在任务卡里写输出路径。
- 对写入工具限制目录。
- 用 Git 查看变更。

### 成本失控

处理：

- 高成本 Tool 前加确认。
- 设置单项目生成次数。
- 失败后先复盘，不要连续重试。

## 24. 本卷总结

MCP 的意义不是让 Agent 无限制做事，而是让它以清楚的方式连接工具。

稳定思路：

```text
先有项目结构
-> 再有上下文资源
-> 再开放必要工具
-> 高风险动作人工确认
-> 全程记录日志
-> 能回滚
```

视频项目里，Agent 最有价值的地方是管理上下文、拆任务、记录过程和辅助生成。最终判断仍然属于创作者。
