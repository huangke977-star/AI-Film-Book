# 第 03 卷：ChatGPT 与 Codex

## 学习目标

本卷解决一个关键问题：什么时候用 ChatGPT，什么时候用 Codex。

读完之后，你应该能做到：

- 区分“聊天助手”和“项目 Agent”的不同用途。
- 知道 ChatGPT 更适合创意讨论、解释、改写和快速问答。
- 知道 Codex 更适合读取项目文件、维护上下文、批量生成、修改文档和执行命令。
- 学会让 Codex 先读 `brief.md`、`storyboard.md`、`style_guide.md`，再生成提示词。
- 用 Codex 写 brief、脚本、分镜、镜头提示词和失败复盘。
- 避免让 Codex 编造平台价格、隐藏能力、账号状态和未核验工具细节。

本卷不要求你会编程。这里的 Codex 用法，是把它当作 AI 视频项目助理，而不是只把它当程序员工具。

## 本卷导读：聊天助手和项目 Agent 的区别

如果你完全不懂 AI 视频，最容易把所有 AI 工具都当成“聊天框”。你可能会打开 ChatGPT 或 Codex，然后说：

```text
帮我写一个 AI 视频提示词。
```

它确实会给你一段文字。但这不是稳定工作流。

稳定工作流应该是：

```text
先有项目文件
-> 让 Codex 读取文件
-> 检查缺失信息
-> 分阶段生成内容
-> 把结果写回文件
-> 生成素材后记录失败
-> 再让 Codex 局部修改
```

ChatGPT 更像一个可以随时讨论的创意顾问；Codex 更像一个能进入项目文件夹工作的制片助理。两者都重要，但不要混用。

## 1. ChatGPT 适合做什么

ChatGPT 适合处理开放式、对话式、解释式任务。

### 1.1 想创意

例如：

```text
我想做一个面向 AI 视频新手的抖音账号，请帮我想 20 个选题方向。
```

ChatGPT 很适合发散，因为它不需要马上改文件。

### 1.2 解释概念

例如：

```text
请用零基础能听懂的方式解释“图生视频”和“文生视频”的区别。
```

### 1.3 改写口播稿

例如：

```text
这段口播太书面，请改成抖音短视频风格，保留核心意思，但更口语。
```

### 1.4 做方向讨论

例如：

```text
我想做商业广告案例和短剧案例，哪个更适合先学？
```

ChatGPT 的优势是轻便、快速、适合讨论。但它的弱点是：如果没有文件上下文，长项目很容易丢状态。

## 2. Codex 适合做什么

Codex 适合处理“项目化任务”。

OpenAI 官方 Codex CLI 文档将 Codex 描述为可以在选定目录中检查文件、编辑文件并运行本机已有工具的编码 Agent。对 AI 视频教程来说，这个能力不只用于代码，也适合 Markdown 教程、Prompt 库、项目模板和工作流文件。

Codex 适合：

- 阅读当前项目目录。
- 生成或修改 Markdown 文件。
- 批量创建 `prompts/shot_01.md` 到 `prompts/shot_05.md`。
- 对比旧版和新版提示词。
- 运行 Git 命令。
- 搜索项目中的术语。
- 统计字数或文件数量。
- 把失败样本整理进 `iteration_log.md`。

Codex 的关键价值是：它能和文件一起工作。

## 3. 为什么视频项目需要文件上下文

一条视频项目至少包含：

```text
brief.md
script.md
storyboard.md
style_guide.md
character_bible.md
prompts/
iteration_log.md
```

如果你只在聊天框里说：

```text
继续帮我改一下。
```

AI 可能不知道你要改的是脚本、分镜、提示词，还是风格。即使它猜对了，也可能忘记前面的约束。

文件上下文的好处是：

- 每个信息有固定位置。
- 可以被 Codex 重新读取。
- 可以被 Git 记录。
- 可以被人类检查。
- 可以被复制到下一个项目。

这就是为什么本书把 AI 视频制作称为“项目工程”，而不是“提示词技巧”。

## 4. Codex 在 AI 视频制作中的 12 个典型任务

| 编号 | 任务 | 输入 | 输出 |
| --- | --- | --- | --- |
| 1 | 项目目录检查 | 当前文件夹 | 缺失文件清单 |
| 2 | Brief 扩写 | 一句话想法 | `brief.md` |
| 3 | 脚本生成 | `brief.md` | `script.md` |
| 4 | 分镜生成 | `brief.md` + `script.md` | `storyboard.md` |
| 5 | 风格手册生成 | brief + 参考方向 | `style_guide.md` |
| 6 | 角色圣经生成 | 人物设定 | `character_bible.md` |
| 7 | 镜头提示词生成 | 分镜 + 风格手册 | `prompts/shot_xx.md` |
| 8 | 失败样本分析 | `iteration_log.md` | 修改建议 |
| 9 | 提示词局部修正 | 失败原因 | 更新单个 prompt |
| 10 | 发布文案生成 | 成片说明 | `publish/copy.md` |
| 11 | Git 提交整理 | 文件改动 | 提交信息建议 |
| 12 | 项目复盘 | 全部项目文件 | 成功经验和下次改进 |

这 12 个任务里，最重要的是第 1 个：项目目录检查。很多失败不是模型问题，而是文件缺失。

## 5. 给 Codex 下任务的基本格式

给 Codex 的指令建议包含 5 个部分：

```text
角色：
输入文件：
任务：
输出格式：
限制：
```

示例：

```text
角色：你是我的 AI 视频项目制片助理。
输入文件：请读取 brief.md、storyboard.md、style_guide.md。
任务：检查当前项目是否可以进入镜头提示词生成阶段。
输出格式：用表格列出已具备信息、缺失信息、风险点。
限制：不要生成提示词，不要编造不存在的角色和场景。
```

这个格式比“帮我看看”稳定得多。

## 6. 让 Codex 先读文件再生成

新手最常犯的错误，是让 Codex 直接输出结果。

不推荐：

```text
帮我生成 5 个 AI 视频提示词。
```

推荐：

```text
请先读取 brief.md、storyboard.md、style_guide.md。
不要马上生成提示词。
请先判断这三个文件是否足够支持 5 个镜头提示词生成。
如果信息不足，请列出缺失项，并给出建议默认值。
```

为什么要这样？

因为 Codex 如果不先检查文件，就可能：

- 引入新的角色。
- 改变风格。
- 忽略时长。
- 忽略平台画幅。
- 生成和分镜无关的镜头。

让 Codex 先读文件，是控制质量的第一步。

## 7. 用 Codex 写 brief、脚本和分镜

### 7.1 写 brief

指令：

```text
你是我的 AI 视频项目制片助理。
我要做一条 15 秒 AI 视频，主题是“用 AI 做第一条短片的过程”。
目标观众是完全没做过 AI 视频的新手。
请创建 brief.md。

brief.md 必须包含：
1. 一句话目标
2. 目标观众
3. 核心观点
4. 情绪关键词
5. 平台和时长
6. 风格边界
7. 禁止出现
```

### 7.2 写脚本

指令：

```text
请读取 brief.md。
为一条 15 秒竖屏短视频生成 script.md。
要求用表格输出，包含时间、画面、字幕、旁白、目的。
不要写成长文章，不要超过 5 个镜头。
```

### 7.3 写分镜

指令：

```text
请读取 brief.md 和 script.md。
生成 storyboard.md。
每个镜头必须包含：镜头编号、时长、画面、主体、动作、景别、运镜、光线、声音、镜头目的。
要求总时长 15 秒，镜头数量 5 个。
```

## 8. 用 Codex 生成镜头提示词

镜头提示词要从分镜来，不要从灵感来。

指令：

```text
请读取 brief.md、storyboard.md、style_guide.md。
为每个镜头生成单独提示词文件：
prompts/shot_01.md
prompts/shot_02.md
prompts/shot_03.md
prompts/shot_04.md
prompts/shot_05.md

每个文件必须包含：
1. 镜头目的
2. 中文提示词
3. English Prompt
4. Negative Prompt
5. 参数建议
6. 可能失败点

限制：
- 不要新增 storyboard.md 之外的镜头。
- 不要改变 style_guide.md 的视觉方向。
- 不要要求视频模型生成可读中文字幕。
```

注意最后一条：不要让视频模型生成可读字幕。字幕应该在后期加。

## 9. 用 Codex 维护失败样本和迭代记录

生成素材后，你应该把问题写进 `iteration_log.md`，再让 Codex 分析。

失败记录示例：

```markdown
## 2026-07-10 第一轮生成

| 镜头 | 文件 | 问题 | 是否可用 | 下一步 |
| --- | --- | --- | --- | --- |
| S01 | S01_V01_raw.mp4 | 屏幕文字乱码 | 否 | 弱化屏幕文字，后期添加字幕 |
| S02 | S02_V01_raw.mp4 | 人物手部变形 | 部分可用 | 避免手部特写 |
| S03 | S03_V01_raw.mp4 | 镜头运动太快 | 否 | 改为固定机位或慢推 |
```

给 Codex 的指令：

```text
请读取 iteration_log.md 和 prompts/。
只修改标记为“否”或“部分可用”的镜头提示词。
不要修改已经可用的镜头。
每个修改都要说明原因，并追加到 iteration_log.md。
```

这比“重新帮我写一版”稳定，因为它只修局部。

## 10. 用 Codex 管理 Git 提交

Codex 可以帮助你整理提交，但你要保持清醒：提交前要看 `git status` 和 diff。

推荐指令：

```text
请检查当前 Git 改动，按文件说明这次改动属于哪个阶段。
不要提交。
先建议一个 commit message，并提醒我哪些文件不应该提交。
```

如果确认无误，再让 Codex 提交：

```text
请提交当前文档改动。
提交信息使用：Add first storyboard and prompts
不要提交 outputs/raw、outputs/final、.env 或任何媒体文件。
```

Git 在视频项目里不是为了炫技，而是为了保留版本。

## 11. 如何避免 Codex 编造工具能力

Codex 很强，但它不是平台后台。它不知道你的真实账号余额，不知道 LibTV 当天价格，不知道某个模型是否刚刚更新，除非你提供可核验资料或它能访问当前页面。

不要这样问：

```text
LibTV 现在最便宜的模型多少钱？
```

更好的问法：

```text
我会自己查看 LibTV 当前页面价格。
请你先帮我设计一个 cost_log.md 模板，用于记录模型、次数、单价、总价和备注。
```

不要这样问：

```text
帮我写一个一定能生成 Ross 同款视频的提示词。
```

更好的问法：

```text
请根据 Ross 案例拆解，设计一个可复刻的项目结构和镜头提示词草案。
所有未核验的具体工具参数都标注为需以实际平台为准。
```

核心原则：

```text
让 Codex 处理项目结构和文本，不让它假装知道未核验的平台事实。
```

## 12. Ross 案例中的 Codex 角色

Ross 案例标题说“全程用 Codex 和 LibTV 完成”。从视频表现看，Codex 很可能不是直接生成画面的工具，而是承担了工作流组织角色。

可以复原为：

| 阶段 | Codex 可能做的事 |
| --- | --- |
| 选题 | 把“手搓提示词时代过去了”扩成视频观点 |
| 脚本 | 组织开场、虚拟形象、成本、模型选择、结语 |
| 分镜 | 拆出每个画面段落 |
| 风格 | 统一电影感、黄色字幕、口播人物、背景切换 |
| 提示词 | 为每个镜头写生成提示词 |
| 复盘 | 根据失败画面修改提示词 |
| 发布 | 写标题、描述、标签 |

这就是本书要你学 Codex 的原因：不是为了追一个工具名，而是为了把创作过程变成可维护系统。

## 13. 可复制 Codex 指令库

这一节可以直接复制使用。第一次学习时不要急着改得很复杂，先按模板跑通，再慢慢替换主题和风格。

### 13.1 项目体检指令

用途：开始任何生成前，先让 Codex 检查项目是否完整。

```text
你是我的 AI 视频项目制片助理。
请读取当前目录下的 brief.md、script.md、storyboard.md、style_guide.md、character_bible.md、iteration_log.md。

任务：
1. 列出已经存在的文件。
2. 列出缺失文件。
3. 判断当前项目是否可以进入镜头提示词生成阶段。
4. 列出 5 个最可能导致生成失败的风险点。

限制：
- 不要生成提示词。
- 不要新增角色、场景或风格。
- 不要编造工具价格和平台能力。

输出格式：
用 Markdown 表格输出“项目状态、缺失信息、风险、建议动作”。
```

### 13.2 Brief 扩写指令

用途：把一句话想法扩成项目目标。

```text
你是我的 AI 视频项目制片助理。
我现在只有一句话想法：
“用 AI 展示一个新手从零开始做第一条视频的过程。”

请创建 brief.md。

brief.md 必须包含：
1. 一句话目标
2. 目标观众
3. 核心观点
4. 情绪关键词
5. 平台和时长
6. 画幅
7. 风格边界
8. 禁止出现的内容

要求：
- 面向零基础新手。
- 不要使用复杂电影术语。
- 不要写成营销文案。
```

### 13.3 分镜生成指令

用途：把项目目标变成 5 个镜头。

```text
请读取 brief.md 和 script.md。
生成 storyboard.md。

要求：
1. 总时长 15 秒。
2. 只生成 5 个镜头。
3. 每个镜头包含：镜头编号、时长、画面、主体、动作、景别、运镜、光线、声音、镜头目的。
4. 每个镜头只表达一个目的。
5. 不要新增 brief.md 之外的主题和角色。

输出格式：
Markdown 表格。
```

### 13.4 风格手册指令

用途：让多个镜头保持统一。

```text
请读取 brief.md 和 storyboard.md。
创建 style_guide.md。

必须包含：
1. 画幅
2. 色彩
3. 光线
4. 镜头质感
5. 固定元素
6. 可变化元素
7. 禁止项

风格方向：
真实纪录片质感，低饱和，冷暖对比，适合新手 AI 视频教程。

限制：
- 不要写“电影感”这种单独空泛词。
- 每个风格描述都要能转成提示词。
```

### 13.5 镜头提示词生成指令

用途：把分镜变成可投喂生成平台的提示词。

```text
请读取 brief.md、storyboard.md、style_guide.md。
为 storyboard.md 中每个镜头生成单独提示词文件。

保存路径：
prompts/shot_01.md
prompts/shot_02.md
prompts/shot_03.md
prompts/shot_04.md
prompts/shot_05.md

每个文件包含：
1. 镜头目的
2. 中文提示词
3. English Prompt
4. Negative Prompt
5. 参数建议
6. 可能失败点

限制：
- 不要要求模型生成可读中文。
- 不要改变 style_guide.md。
- 不要生成 storyboard.md 之外的镜头。
```

### 13.6 第一轮失败复盘指令

用途：生成素材后，让 Codex 只修改失败镜头。

```text
请读取 iteration_log.md 和 prompts/ 目录。

任务：
1. 找出 iteration_log.md 中标记为“否”或“部分可用”的镜头。
2. 分析每个失败原因属于：需求问题、提示词问题、素材问题、模型问题、后期问题。
3. 只修改对应镜头的 prompt 文件。
4. 不修改已经可用的镜头。
5. 把修改原因追加到 iteration_log.md。

限制：
- 不要重写整个项目。
- 不要改变已经确认的整体风格。
- 不要把字幕交给视频模型生成。
```

### 13.7 发布文案指令

用途：成片完成后，生成平台文案。

```text
请读取 brief.md、storyboard.md、iteration_log.md。
为这个 15 秒 AI 视频生成发布文案。

输出：
1. 5 个抖音标题。
2. 3 个小红书标题。
3. 1 段抖音发布文案。
4. 1 段小红书发布文案。
5. 10 个话题标签。

限制：
- 不夸大 AI 能力。
- 不承诺一键生成大片。
- 强调“从零开始跑通流程”。
```

### 13.8 项目复盘指令

用途：项目结束后，把经验沉淀下来。

```text
请读取 brief.md、storyboard.md、prompts/、iteration_log.md。
生成 project_review.md。

必须包含：
1. 项目目标是否达成。
2. 最成功的镜头。
3. 最失败的镜头。
4. 最有复用价值的提示词。
5. 最大的成本浪费。
6. 下次项目应该保留的规则。
7. 下次项目应该避免的问题。

要求：
- 写得具体。
- 不要只写“效果不错”。
- 每条结论都要对应一个文件或镜头。
```

## 14. 一个 15 秒项目中的 Codex 协作流程

下面是一个完整示例，展示 Codex 如何参与一个小项目。

### 14.1 初始想法

```text
我想做一条 15 秒视频，展示 AI 视频新手如何从零开始。
```

如果直接拿这句话去生成视频，结果会很随机。正确做法是让 Codex 先拆项目。

### 14.2 第一次让 Codex 做什么

```text
你是我的 AI 视频项目制片助理。
请根据下面的一句话想法创建 brief.md：
“我想做一条 15 秒视频，展示 AI 视频新手如何从零开始。”

要求：
- 面向完全新手。
- 平台是抖音。
- 画幅是 9:16。
- 核心观点是“先跑通流程，再追求大片”。
```

输出不是最终视频，而是 `brief.md`。

### 14.3 第二次让 Codex 做什么

```text
请读取 brief.md。
生成 15 秒视频脚本 script.md。
脚本分成 5 个镜头，每个镜头 3 秒。
输出表格：时间、画面、字幕、旁白、目的。
```

### 14.4 第三次让 Codex 做什么

```text
请读取 brief.md 和 script.md。
生成 storyboard.md。
每个镜头必须包含主体、动作、景别、运镜和光线。
```

### 14.5 第四次让 Codex 做什么

```text
请读取 brief.md、script.md、storyboard.md。
创建 style_guide.md。
要求让 5 个镜头看起来属于同一个短片。
```

### 14.6 第五次让 Codex 做什么

```text
请读取 brief.md、storyboard.md、style_guide.md。
为每个镜头生成 prompts/shot_01.md 到 prompts/shot_05.md。
不要让视频模型生成字幕，字幕留给后期。
```

### 14.7 生成后让 Codex 做什么

生成后不要只说“效果不好”。要写成结构化记录：

```markdown
| 镜头 | 问题 | 是否可用 | 下一步 |
| --- | --- | --- | --- |
| S01 | 画面还可以，但屏幕文字乱码 | 部分可用 | 弱化屏幕文字 |
| S02 | 镜头运动太快 | 否 | 改成固定机位 |
| S03 | 可以使用 | 是 | 保留 |
```

然后给 Codex：

```text
请读取 iteration_log.md。
只修改 S01 和 S02 的提示词。
S03 已经可用，不要修改。
```

这个流程看起来慢，但它能避免你在失败后彻底重写项目。

## 15. Codex 输出质量检查表

Codex 给出结果后，不要马上接受。用这张表检查：

| 检查项 | 问题 |
| --- | --- |
| 是否读取了指定文件 | 如果没有，结果可能脱离项目 |
| 是否改变了主题 | 改主题会导致项目漂移 |
| 是否新增了角色 | 新角色会增加一致性风险 |
| 是否改变了风格 | 风格漂移会让镜头不像同一部片 |
| 是否包含未核验事实 | 价格、权限、模型能力需要核验 |
| 是否输出到正确文件 | 聊天内容不等于项目文件 |
| 是否修改了不该改的文件 | 批量修改前后都要看 Git 状态 |
| 是否给出失败风险 | 没有风险提示，下一轮复盘会困难 |

这张表的目的不是不信任 Codex，而是把它当成协作者。协作就需要检查。

## 16. 进入第 04 卷前的检查清单

第 04 卷会讲 Context Engineering，也就是把单条提示词升级成项目上下文。进入下一卷前，你应该先确认自己已经理解 Codex 的基本协作方式。

```markdown
| 检查项 | 是否能做到 |
| --- | --- |
| 我能说清楚 ChatGPT 和 Codex 的分工 |  |
| 我知道 Codex 应该先读项目文件再生成 |  |
| 我能写出一条项目体检指令 |  |
| 我能让 Codex 生成 brief、script、storyboard |  |
| 我知道每个镜头提示词应该单独保存 |  |
| 我知道失败样本要写入 iteration_log.md |  |
| 我知道实时价格和平台权限不能让 Codex 编造 |  |
| 我会在提交前检查 Git 状态 |  |
```

如果这张表里还有很多空白，先不要急着学习 Context Engineering。因为第 04 卷会默认你已经接受一个前提：AI 视频项目不是一次聊天，而是一组文件共同构成的上下文系统。

换句话说，第 03 卷让你学会“怎样指挥 Codex”，第 04 卷会进一步解释“Codex 应该围绕哪些上下文文件工作”。这两个能力连起来，才是从手搓提示词走向项目化创作的关键。

先稳住协作方式，再追求自动化。这一步很重要。

## 17. 常见错误

### 错误 1：把 Codex 当普通聊天框

表现：

- 不让它读文件。
- 不告诉它输出路径。
- 不要求它保留已有结构。

修正：

明确输入文件、输出文件和限制。

### 错误 2：一次让 Codex 做太多事

表现：

- 同时要求写脚本、分镜、提示词、剪辑方案和发布文案。

修正：

分阶段：brief、script、storyboard、prompts、iteration。

### 错误 3：让 Codex 替你判断未核验事实

表现：

- 询问实时价格。
- 询问账号权限。
- 询问平台隐藏能力。

修正：

实时事实自己核验，Codex 负责记录和组织。

### 错误 4：不检查 Codex 改了什么

表现：

- 让 Codex 批量改文件后直接提交。

修正：

提交前看 `git status` 和 diff，尤其确认没有提交密钥和媒体。

## 18. 故障排查

### Codex 生成内容跑题

检查：

- 是否提供了 brief。
- 是否让它先读文件。
- 是否明确禁止新增角色和场景。

处理：

```text
请回到 brief.md 和 storyboard.md。
撤回刚才新增的角色和场景。
只保留原分镜中的 5 个镜头。
```

### Codex 输出太长

检查：

- 是否要求表格或固定格式。
- 是否限定镜头数量。

处理：

```text
请把上面的内容压缩成表格，每个镜头不超过 80 个字。
```

### Codex 修改了不该改的文件

处理：

1. 用 Git 查看改动。
2. 保留需要的文件。
3. 对不该改的文件恢复。

给 Codex 的指令：

```text
请说明这次修改涉及哪些文件。
如果某个文件和当前任务无关，请不要继续修改它。
```

### Codex 开始编造平台能力

处理：

```text
请停止使用未核验平台事实。
把具体价格、模型名称、按钮路径全部改成“以实际平台为准”。
只保留稳定工作流建议。
```

## 19. 本卷总结

ChatGPT 和 Codex 都有价值，但分工不同。

```text
ChatGPT：适合讨论、解释、创意、改写。
Codex：适合读文件、改文件、批量生成、维护上下文、执行项目任务。
```

AI 视频项目越复杂，越需要 Codex 这类项目 Agent。它能帮助你把创意变成 brief，把 brief 变成脚本，把脚本变成分镜，把分镜变成提示词，再把失败样本变成下一轮改进。

但 Codex 不是万能的。涉及实时价格、平台权限、账号状态、模型更新等事实，必须核验。涉及创作判断、镜头取舍、商业目标，也仍然需要人类决定。

## 20. 课后练习

### 练习 1：写一条 Codex 项目检查指令

要求它读取：

```text
brief.md
storyboard.md
style_guide.md
```

输出：

```text
已具备信息
缺失信息
生成风险
下一步建议
```

### 练习 2：写一条分镜生成指令

要求：

- 总时长 15 秒。
- 5 个镜头。
- 每个镜头包含目的。
- 不新增 brief 之外的角色。

### 练习 3：写一条失败样本修正指令

输入：

```text
S01 屏幕文字乱码。
S02 人物手部变形。
S03 镜头运动太快。
```

要求 Codex：

- 只修改失败镜头。
- 不改变整体风格。
- 把修改理由写入 `iteration_log.md`。

### 练习 4：写一条防编造指令

要求 Codex：

- 不写实时价格。
- 不写未核验模型能力。
- 不写账号权限。
- 所有工具细节标注“以实际平台为准”。
