# 第 02 卷：环境搭建

## 学习目标

本卷解决一个非常实际的问题：如何把 AI 视频创作从“临时聊天记录”变成“可维护项目”。

读完之后，你应该能做到：

- 建立一个标准 AI 视频项目目录。
- 知道 `brief.md`、`script.md`、`storyboard.md`、`style_guide.md`、`prompts/`、`outputs/` 分别负责什么。
- 用 Git 记录创作过程，而不是只把 Git 当成上传 GitHub 的工具。
- 设计清晰的素材命名规则，避免生成文件越来越乱。
- 知道哪些文件可以提交，哪些文件不能提交。
- 准备 Codex、LibTV 或同类生成平台、剪辑软件的基本工作区。

本卷不是电脑装机教程，而是创作工程的搭建教程。你不需要懂编程，也可以按这里的规则管理 AI 视频项目。

## 本卷导读：先把创作放进文件夹

很多新手的问题不是不会写提示词，而是所有东西都散落在不同地方：

- 想法在聊天记录里。
- 分镜在临时文档里。
- 提示词在剪贴板里。
- 生成视频在下载目录里。
- 可用素材和失败素材混在一起。
- 过几天再看，完全不知道哪一版能用。

这会让 AI 视频变成一次性实验。今天运气好生成了一个好镜头，明天想复用却找不到原始提示词；今天失败了十次，过几天忘了失败原因，又重复花钱试错。

环境搭建的目的，就是让每一次创作都留下结构化痕迹。

```text
想法有 brief
脚本有 script
镜头有 storyboard
风格有 style guide
提示词有 prompts
素材有 outputs
失败有 iteration log
版本有 Git
```

当这些东西稳定下来，Codex 才能真正帮你工作。因为 Codex 最擅长的是读取文件、维护上下文、批量生成和修正，而不是从一段混乱聊天里猜你的项目状态。

## 1. 为什么环境搭建是创作能力的一部分

传统视频制作里，剧组会有分工：导演、制片、摄影、灯光、美术、场记、剪辑。AI 视频创作者往往一个人承担很多角色，所以更需要文件系统帮你保持秩序。

环境搭建解决四个问题：

### 1.1 可复用

如果你的提示词、风格、角色设定都保存在文件里，下次做类似项目可以直接复制修改。

### 1.2 可复盘

失败不是浪费。失败样本记录清楚后，下一轮就知道该改提示词、换参考图、换模型，还是后期处理。

### 1.3 可协作

以后你把项目交给剪辑师、设计师、客户或另一个 AI 助手时，对方能从文件看懂项目，而不是翻你的聊天记录。

### 1.4 可长期维护

本书目标是长期维护的百科教程，你自己的作品项目也应该长期维护。一个好项目不只是成片，还包括能解释成片如何产生的全部材料。

## 2. 推荐电脑和软件准备

AI 视频制作不一定需要顶级电脑，因为大量生成发生在云端。但你至少需要一个稳定的本地工作环境。

### 2.1 基础准备

建议准备：

- 一个固定项目目录，例如 `D:\AIvideo\projects`。
- 一个文本编辑器，例如 VS Code、Typora、Obsidian 或其他 Markdown 编辑器。
- Git。
- Codex 或能读取本地项目文件的 Agent 工具。
- 一个视频生成平台，例如 LibTV 或同类平台。
- 一个剪辑工具，例如剪映、Premiere Pro、DaVinci Resolve、Final Cut Pro。

### 2.2 新手不必一开始准备什么

不必一开始准备：

- 多个 API Key。
- 自动化脚本。
- MCP 服务。
- n8n 工作流。
- 专业调色系统。
- 大量插件。

第一阶段只需要能管理文件、能生成素材、能剪辑导出即可。

## 3. Git 的作用：记录创作而不是只上传代码

很多非程序员听到 Git 会紧张，以为这是程序员工具。其实 Git 在 AI 视频项目里可以很简单：

```text
记录每个阶段的文件变化。
保存一版稳定状态。
必要时回看之前的提示词和分镜。
把教程或项目同步到 GitHub。
```

你可以把 Git 想成“项目时间机器”。

### 3.1 最常用命令

| 命令 | 作用 | 什么时候用 |
| --- | --- | --- |
| `git status` | 看哪些文件变了 | 每次提交前 |
| `git add .` | 把当前改动放进提交区 | 确认要保存这些改动时 |
| `git commit -m "message"` | 保存一个版本 | 完成一个阶段时 |
| `git log --oneline` | 查看历史提交 | 想回看进度时 |
| `git push` | 推送到远程仓库 | 本地提交后同步 GitHub |

### 3.2 AI 视频项目的提交节奏

建议这些节点提交：

- 创建项目目录后。
- 写完 brief 后。
- 写完分镜后。
- 写完第一批提示词后。
- 完成第一轮生成复盘后。
- 完成成片后。

提交信息可以简单清楚：

```bash
git commit -m "Create project brief"
git commit -m "Add first storyboard"
git commit -m "Add shot prompts"
git commit -m "Record first generation review"
```

不需要写得花哨，关键是让未来的你看得懂。

## 4. AI 视频项目目录结构

每个视频项目都应该单独建一个文件夹。不要把所有作品混在一个大目录里。

推荐结构：

```text
2026-07-ross-style-demo/
├── brief.md
├── script.md
├── storyboard.md
├── style_guide.md
├── character_bible.md
├── prompts/
│   ├── shot_01.md
│   ├── shot_02.md
│   ├── shot_03.md
│   ├── shot_04.md
│   └── shot_05.md
├── outputs/
│   ├── raw/
│   ├── selected/
│   ├── rejected/
│   └── final/
├── edit/
├── references/
├── publish/
├── cost_log.md
└── iteration_log.md
```

### 4.1 文件夹职责

| 路径 | 作用 |
| --- | --- |
| `brief.md` | 项目目标、观众、平台、风格边界 |
| `script.md` | 口播稿、字幕稿或信息节奏 |
| `storyboard.md` | 每个镜头的画面、时长、目的 |
| `style_guide.md` | 画面色彩、光线、质感、禁止项 |
| `character_bible.md` | 固定人物的外观、服装、语气、不能变化的特征 |
| `prompts/` | 每个镜头单独提示词 |
| `outputs/raw/` | 原始生成结果 |
| `outputs/selected/` | 可进入剪辑的素材 |
| `outputs/rejected/` | 有代表性的失败样本 |
| `outputs/final/` | 导出的终版视频 |
| `edit/` | 剪辑工程文件 |
| `references/` | 参考图、参考视频、客户资料 |
| `publish/` | 标题、封面、发布文案、话题标签 |
| `cost_log.md` | 成本记录 |
| `iteration_log.md` | 失败和修改记录 |

## 5. Brief、Script、Storyboard、Style Guide、Prompts、Outputs 的职责

这些文件不是形式主义。它们负责把复杂项目拆成可理解的小块。

### 5.1 brief.md

`brief.md` 是方向盘。

模板：

```markdown
# Brief

## 一句话目标

## 目标观众

## 核心观点

## 情绪关键词

## 平台和时长

## 风格边界

## 禁止出现
```

### 5.2 script.md

`script.md` 决定观众听到或看到的信息顺序。

短视频脚本可以分成：

```text
开场钩子
过程展示
结果证明
结尾观点
```

### 5.3 storyboard.md

`storyboard.md` 是工程图。它把脚本变成镜头。

推荐字段：

```markdown
| 镜头 | 时长 | 画面 | 主体 | 动作 | 景别 | 运镜 | 光线 | 声音 | 目的 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |
```

### 5.4 style_guide.md

`style_guide.md` 负责统一画面。

它应该写清楚：

- 画幅。
- 色彩。
- 光线。
- 镜头质感。
- 固定元素。
- 可变化元素。
- 禁止项。

### 5.5 prompts/

`prompts/` 里每个镜头一个文件。不要把所有镜头写进同一个提示词。

示例：

```text
prompts/shot_01.md
prompts/shot_02.md
prompts/shot_03.md
```

这样做的好处是：如果 S03 失败，只改 S03，不影响其他镜头。

### 5.6 outputs/

`outputs/` 负责保存生成结果。不要把素材只放在系统下载目录。

原始输出、筛选素材、失败素材、终版成片应该分开。

## 6. 素材命名规范

AI 视频生成会产生大量文件，命名混乱会立刻毁掉项目。

推荐格式：

```text
镜头号_版本号_状态_简短说明.ext
```

示例：

```text
S01_V01_raw_workspace_opening.mp4
S01_V02_selected_workspace_opening.mp4
S02_V03_rejected_text_garbled.mp4
S05_V01_final_phone_playback.mp4
```

### 6.1 状态定义

| 状态 | 含义 |
| --- | --- |
| `raw` | 原始生成素材 |
| `selected` | 可进入剪辑 |
| `rejected` | 失败样本 |
| `final` | 最终使用版本 |

### 6.2 为什么保留 rejected

失败样本不是垃圾。它能告诉你：

- 哪种提示词容易导致变脸。
- 哪种动作模型不擅长。
- 哪种场景会导致背景漂移。
- 哪种文字要求会变乱码。

你不需要保存每一个失败版本，但要保存有代表性的失败版本。

## 7. `.env`、密钥和账号安全

AI 视频项目可能涉及 API Key、账号 Token、Cookie、下载链接和付费平台信息。安全规则很简单：

```text
真实密钥只放本地，不写进 Markdown，不提交 Git。
```

### 7.1 可以提交的文件

| 文件 | 是否可提交 | 说明 |
| --- | --- | --- |
| `brief.md` | 可以 | 不含隐私时可提交 |
| `storyboard.md` | 可以 | 项目创作文件 |
| `style_guide.md` | 可以 | 风格说明 |
| `prompts/shot_01.md` | 可以 | 不含密钥时可提交 |
| `iteration_log.md` | 可以 | 注意不要写账号信息 |

### 7.2 不应该提交的文件

| 文件 | 原因 |
| --- | --- |
| `.env` | 可能包含 API Key |
| `*.token` | 可能包含登录令牌 |
| `*.cookie` | 可能包含登录状态 |
| 下载的视频素材 | 文件大，版权和体积都不适合 |
| 客户原始资料 | 可能包含隐私或商业信息 |

### 7.3 `.gitignore` 示例

```text
.env
*.token
*.cookie
outputs/raw/
outputs/final/
*.mp4
*.mov
```

本书仓库已经忽略 `.analysis/` 和 `.worktrees/`，避免把临时分析文件和本地工作树提交进去。

## 8. Codex 工作区准备

Codex 最适合处理“有文件、有上下文、有步骤”的任务。准备 Codex 工作区时，记住三条原则。

### 8.1 让 Codex 先读文件

不要直接说：

```text
帮我写提示词。
```

更好的说法：

```text
请先阅读当前目录下的 brief.md、storyboard.md、style_guide.md。
不要立刻生成提示词。
先告诉我这些文件里缺少哪些信息，以及哪些信息可能导致生成失败。
```

### 8.2 一次只让 Codex 做一个阶段

不要一次要求：

```text
帮我写脚本、分镜、提示词、生成视频、剪辑、发布。
```

更稳的方式：

```text
第一步：检查 brief。
第二步：生成分镜。
第三步：生成每个镜头提示词。
第四步：根据失败样本修改提示词。
```

### 8.3 把输出落到文件

只留在聊天里的内容很容易丢。要求 Codex 修改或创建文件：

```text
请把结果写入 storyboard.md。
请把每个镜头提示词分别保存到 prompts/shot_01.md 到 prompts/shot_05.md。
请把失败原因追加到 iteration_log.md。
```

## 9. LibTV 或同类平台准备

在本书里，LibTV 代表“生成素材的平台或工作流入口”。它不负责替你想清楚项目，也不负责最终剪辑。

你在打开平台前，应该已经准备好：

- `brief.md`
- `storyboard.md`
- `style_guide.md`
- `prompts/shot_01.md`
- 输出目录 `outputs/raw/`
- 失败记录 `iteration_log.md`

### 9.1 新手先用手动复制

第一阶段建议先用最朴素的方式：

```text
Codex 生成提示词
-> 手动复制到 LibTV 或同类平台
-> 下载结果到 outputs/raw/
-> 人工筛选
-> 把问题写入 iteration_log.md
-> 让 Codex 修改对应提示词
```

Skill、Agent、API、MCP 都可以后面再学。先用手动方式能让你真正理解输入和输出。

### 9.2 生成前检查

每次生成前确认：

| 项目 | 是否明确 |
| --- | --- |
| 画幅 | 9:16、16:9 或其他 |
| 时长 | 3 秒、5 秒或其他 |
| 输入类型 | 文生视频、图生视频、视频生视频 |
| 主体 | 谁或什么在画面中 |
| 动作 | 主体做什么 |
| 风格 | 是否引用 style guide |
| 负面约束 | 是否写清楚避免项 |
| 输出路径 | 是否知道下载后放哪里 |

## 10. 剪辑软件和导出目录准备

AI 生成素材不是成片。你还需要剪辑。

剪辑阶段通常做：

- 选择素材。
- 调整顺序。
- 控制节奏。
- 添加字幕。
- 添加配乐和音效。
- 调整颜色。
- 导出成片。

建议建立：

```text
edit/
outputs/final/
publish/
```

`edit/` 放剪辑工程，`outputs/final/` 放导出视频，`publish/` 放标题、封面和文案。

## 12. 常见错误

### 错误 1：所有项目放一个文件夹

后果：素材混乱、提示词混乱、无法复盘。

修正：每个视频一个独立项目目录。

### 错误 2：生成素材留在下载目录

后果：过几天找不到，文件名也没有意义。

修正：下载后立即移动到 `outputs/raw/` 并改名。

### 错误 3：把密钥写进文档

后果：提交 GitHub 后可能泄露账号。

修正：真实密钥只放本地 `.env`，公开文档只写字段名。

### 错误 4：没有失败记录

后果：重复犯同样错误。

修正：每轮生成后更新 `iteration_log.md`。

### 错误 5：没有版本提交

后果：改坏后不知道怎么恢复。

修正：每完成一个阶段就 Git commit。

## 13. 故障排查

### Git 显示很多不该提交的素材

处理：

1. 检查 `.gitignore`。
2. 把大视频、临时目录、密钥文件加入忽略。
3. 如果已经 `git add`，先取消暂存。

常用命令：

```bash
git status
git restore --staged <文件名>
```

### Codex 不知道项目背景

处理：

- 确认文件是否存在。
- 指令里明确要求先读哪些文件。
- 不要只给一句“继续”。

示例：

```text
请先读取 brief.md、storyboard.md、style_guide.md，然后总结项目当前状态。
```

### 素材越来越乱

处理：

- 统一命名。
- 每天结束前整理 `outputs/raw/`。
- 把可用素材复制到 `outputs/selected/`。
- 把失败代表样本移到 `outputs/rejected/`。

### 不知道下一步该生成什么

处理：

- 回到 `storyboard.md`。
- 看哪个镜头状态还是未生成。
- 只生成下一个镜头，不要同时开很多任务。

## 14. 环境检查清单

进入第 03 卷前，确认：

```markdown
| 项目 | 是否完成 |
| --- | --- |
| 我知道每个视频要单独建项目目录 |  |
| 我知道 brief、script、storyboard、style guide 的职责 |  |
| 我知道 prompts/ 里每个镜头单独一个文件 |  |
| 我知道 raw、selected、rejected、final 的区别 |  |
| 我知道真实密钥不能提交 Git |  |
| 我会用 git status 查看文件状态 |  |
| 我知道生成素材下载后应该放到哪里 |  |
| 我有一个 iteration_log.md 模板 |  |
```

如果这张表看不懂，先不要进入自动化和 API。先把手动项目目录跑通。

### 14.1 环境搭建验收标准

完成本卷后，你应该能随手创建一个新项目，并在 5 分钟内解释每个文件夹的作用。更具体地说，你应该知道：新想法先写进 `brief.md`，镜头安排写进 `storyboard.md`，统一风格写进 `style_guide.md`，每个镜头提示词放进 `prompts/`，生成素材先进入 `outputs/raw/`，筛选后再进入 `outputs/selected/`，失败原因写进 `iteration_log.md`。

如果你能做到这一点，就说明环境已经能支持后面的 Codex 协作。如果做不到，后面让 Codex 生成提示词时，它也会因为上下文混乱而产出不稳定结果。

## 15. 本卷总结

环境搭建不是杂事，而是 AI 视频创作的基础能力。

一个可靠项目应该至少包含：

```text
brief.md
script.md
storyboard.md
style_guide.md
prompts/
outputs/
iteration_log.md
```

你不需要一开始就自动化，但必须让每一步有文件、有位置、有名字、有记录。

第 01 卷让你知道 AI 视频是什么；第 02 卷让你把这个认知放进一个能长期维护的工作区。下一卷会进入 ChatGPT 与 Codex 的分工，开始让 Agent 真正参与项目。
