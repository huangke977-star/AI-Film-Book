# 第 16 卷：n8n 自动化

## 学习目标

学完本卷后，你应该能做到：

- 理解自动化适合处理什么，不适合处理什么。
- 知道 n8n 工作流、节点、触发器、凭证、执行记录和错误处理的基本概念。
- 能把 AI 视频项目拆成可自动化环节和必须人工判断环节。
- 会设计选题收集、脚本草稿、生成任务清单、下载归档、复盘记录等工作流。
- 知道凭证、API Key、账号权限和日志记录的安全边界。
- 能为批量视频项目建立失败重试、人工终审和成本控制机制。

本卷不教你把创作变成无脑流水线。它教你把重复劳动交给自动化，把关键判断留给人。

## 本卷导读：自动化不是无人生产

学完前 15 卷后，你已经知道一条 AI 视频会产生很多文件：

```text
brief.md
script.md
storyboard.md
prompts/
outputs/
audio/
edit/
delivery/
revision_log.md
```

如果你每次都手动复制、命名、整理、记录，就会很累。自动化的价值是：

```text
减少重复操作
降低漏填文件的概率
让流程可追踪
让批量任务有记录
```

但自动化不应该替你做所有决定。尤其是：

- 是否发布。
- 是否符合品牌。
- 是否侵犯权利。
- 是否误导观众。
- 是否接受客户修改。
- 是否继续投入成本。

这些都需要人判断。

## 1. 自动化适合做什么

适合自动化的任务通常有四个特点：

1. 重复发生。
2. 输入格式稳定。
3. 判断规则清楚。
4. 出错后可以回滚或人工处理。

AI 视频里适合自动化的环节：

| 环节 | 自动化方式 |
| --- | --- |
| 选题收集 | 定时收集表格、RSS、Notion、飞书或表单数据 |
| 项目建档 | 根据模板创建文件夹和 Markdown 文件 |
| 脚本草稿 | 调用 AI 生成第一版，不直接发布 |
| 任务清单 | 把分镜表拆成生成任务 |
| 文件归档 | 下载结果后按规则重命名 |
| 状态通知 | 生成完成或失败时提醒 |
| 成本记录 | 记录模型、次数、日期和结果 |
| 复盘表 | 自动生成复盘模板 |

## 2. 不适合自动化的环节

这些环节不应该完全交给自动化：

- 客户需求判断。
- 广告合规判断。
- 人物和产品是否准确。
- 声音克隆授权确认。
- 音乐商用范围确认。
- 是否接受最终成片。
- 是否发布到账号。

你可以让工作流提醒你检查，但不要让它跳过人工确认。

### 2.1 人工确认点

```markdown
| 节点 | 必须人工确认的问题 |
| --- | --- |
| Brief 完成后 | 客户目标是否清楚 |
| 脚本生成后 | 信息是否准确 |
| 分镜生成后 | 是否服务目标 |
| 视频生成后 | 产品和人物是否正确 |
| 配音音乐后 | 授权是否清楚 |
| 导出前 | 是否可发布 |
```

## 3. n8n 基本概念

n8n 是一个工作流自动化工具。官方文档把 workflow、node、credentials、executions、error handling 等作为核心概念说明。本书只讲 AI 视频项目里的用法。

### 3.1 Workflow

Workflow 是一条自动化流程。

例如：

```text
表单提交选题
-> 创建项目文件夹
-> 生成 brief 草稿
-> 发送给你确认
```

### 3.2 Node

Node 是流程里的一个动作。

常见节点类型：

- 触发器。
- HTTP 请求。
- 表格读写。
- 文件操作。
- AI 模型调用。
- 条件判断。
- 通知。

### 3.3 Trigger

Trigger 是流程开始的条件。

例如：

- 每天早上 9 点执行。
- 表单有新提交。
- 文件夹出现新文件。
- Webhook 收到请求。

### 3.4 Credentials

Credentials 是连接外部服务的凭证。比如 API Key、OAuth、账号授权。凭证不能写进 Markdown，也不能发给不需要的人。

### 3.5 Executions

Executions 是每次工作流运行记录。出问题时，你要看它在哪里失败、输入是什么、输出是什么。

## 4. AI 视频自动化总流程

一个完整但保守的自动化流程可以是：

```text
选题表单
-> 创建项目目录
-> 生成 brief 草稿
-> 人工确认
-> 生成脚本草稿
-> 人工确认
-> 生成分镜表
-> 人工确认
-> 拆生成任务
-> 提醒人工执行或调用平台 API
-> 下载结果
-> 文件归档
-> 生成复盘表
-> 人工终审
```

注意：越靠近发布和商业判断，越要有人确认。

## 5. 节点清单模板

```markdown
| 节点 | 类型 | 输入 | 输出 | 风险 | 人工确认 |
| --- | --- | --- | --- | --- | --- |
| Topic Form | 表单触发 | 选题信息 | 原始需求 | 输入不完整 | 否 |
| Create Project | 文件/脚本 | 项目名 | 项目目录 | 命名冲突 | 否 |
| Draft Brief | AI | 选题 | brief 草稿 | 理解错误 | 是 |
| Draft Script | AI | brief | 脚本草稿 | 信息不准 | 是 |
| Shot List | AI | 脚本 | 分镜表 | 镜头太复杂 | 是 |
| Notify Review | 通知 | 草稿链接 | 审核提醒 | 漏审 | 是 |
```

这张表要先写，再去搭节点。

## 6. 选题收集自动化

选题收集最适合自动化。

来源可以是：

- 表单。
- 表格。
- Notion。
- 飞书或企业微信。
- RSS。
- 手动添加的 Markdown。

### 6.1 选题表字段

```markdown
| 字段 | 说明 |
| --- | --- |
| topic | 选题 |
| audience | 目标观众 |
| platform | 发布平台 |
| goal | 视频目标 |
| source | 来源 |
| priority | 优先级 |
| deadline | 截止时间 |
```

### 6.2 自动生成项目编号

命名示例：

```text
20260710_ai_workflow_douyin_001
```

项目编号可以用于文件夹、任务、导出文件和复盘表。

## 7. 脚本草稿自动化

脚本可以自动生成草稿，但不能直接当最终稿。

### 7.1 输入

```markdown
主题：
目标观众：
视频目标：
平台：
时长：
风格：
禁止内容：
```

### 7.2 输出

```markdown
# script_v01_draft

## 开头钩子

## 中段信息

## 结尾行动

## 风险提醒

## 需要人工确认的问题
```

把“需要人工确认的问题”写出来很重要。自动化不是假装自己永远正确，而是把不确定性暴露出来。

## 8. 生成任务清单自动化

当脚本和分镜确定后，可以把镜头表转成任务卡。

### 8.1 输入分镜表

```markdown
| shot_id | purpose | subject | action | scene | duration | ratio |
| --- | --- | --- | --- | --- | --- | --- |
| S01 | 开头钩子 | 创作者 | 看向屏幕 | 夜晚工作室 | 4s | 9:16 |
```

### 8.2 输出任务卡

```markdown
# S01 Task Card

## Purpose

开头钩子。

## Prompt

夜晚工作室，年轻创作者看向电脑屏幕，屏幕冷光，真实纪录片风格，9:16。

## Negative

不要可读文字，不要水印，不要多余人物。

## Review

检查人物、场景、字幕空间和动作稳定性。
```

自动化可以生成任务卡，但镜头是否合理仍要人看。

## 9. 下载和归档自动化

生成结果如果散落在下载目录，很快就会乱。

自动归档规则：

```text
outputs/raw/
outputs/selected/
outputs/rejected/
```

### 9.1 文件命名规则

```text
SHOT_镜头编号_版本_来源.ext
```

示例：

```text
SHOT_S01_v01_libtv.mp4
SHOT_S01_v02_seedance.mp4
SHOT_S02_v01_kling.mp4
```

### 9.2 归档记录

```markdown
| 文件 | 来源 | 时间 | 状态 | 问题 |
| --- | --- | --- | --- | --- |
| SHOT_S01_v01_libtv.mp4 | LibTV | 2026-07-10 | raw | 人物太远 |
| SHOT_S01_v02_libtv.mp4 | LibTV | 2026-07-10 | selected | 可用 |
```

## 10. 失败重试和人工终审

自动化必须考虑失败。

常见失败：

- API 超时。
- 文件下载失败。
- 输出为空。
- 文件名重复。
- 凭证过期。
- 费用超出预期。
- 平台返回错误。

### 10.1 失败记录

```markdown
| 时间 | 节点 | 错误 | 输入 | 处理 |
| --- | --- | --- | --- | --- |
| 09:31 | Draft Script | 请求超时 | topic_001 | 10 分钟后重试 |
| 10:12 | Download File | 文件不存在 | S03_v01 | 人工检查平台状态 |
```

### 10.2 重试原则

- 可以重试网络错误。
- 可以重试临时超时。
- 不要无限重试同一个失败。
- 成本高的节点要设置上限。
- 涉及发布和客户交付的节点必须人工确认。

## 11. 成本控制

批量任务最怕悄悄烧成本。

### 11.1 成本记录表

```markdown
| 日期 | 项目 | 节点 | 工具 | 次数 | 估算成本 | 结果 |
| --- | --- | --- | --- | --- | --- | --- |
| 07-10 | ai_workflow_001 | video_generation | LibTV | 3 | 待填 | 1 个可用 |
| 07-10 | ai_workflow_001 | tts | TTS | 2 | 待填 | 1 个可用 |
```

### 11.2 成本上限

每个项目应有：

```text
最大生成次数
最大重试次数
最大人工工时
是否需要客户确认后继续
```

没有上限的自动化，会让你失去控制。

## 12. 安全和密钥

自动化工具通常需要连接很多服务。安全规则必须提前写。

### 12.1 基本规则

- API Key 不写进 Markdown。
- 凭证只放在工具的凭证管理里。
- 不把个人账号权限给不需要的人。
- 不把客户素材发送到未确认的平台。
- 不把自动化日志公开。
- 不在测试工作流里使用真实客户数据。

### 12.2 权限表

```markdown
| 服务 | 权限 | 用途 | 风险 | 负责人 |
| --- | --- | --- | --- | --- |
| AI API | 生成文本 | 脚本草稿 | 费用和内容风险 | 我 |
| 表格 | 读写选题 | 任务记录 | 数据泄露风险 | 我 |
| 文件存储 | 上传下载 | 归档素材 | 客户素材风险 | 我 |
```

权限越少越好。能只读就不要给写权限。

## 13. 批量视频工作流案例

假设你要每周做 5 条 AI 视频选题草稿。

### 13.1 流程

```text
每周一 9:00 触发
-> 读取选题表中 priority=high 的 5 条
-> 为每条创建项目目录
-> 生成 brief 草稿
-> 生成 30 秒脚本草稿
-> 生成分镜表草稿
-> 写入 review 队列
-> 通知你逐条确认
```

### 13.2 人工确认后

```text
确认通过
-> 生成任务卡
-> 进入人工或 API 生成
-> 下载归档
-> 生成复盘模板
```

### 13.3 不自动做的事

- 不直接生成最终成片。
- 不直接发布。
- 不替客户确认素材权利。
- 不替你判断广告文案是否合规。

## 14. 数据结构：自动化前先统一字段

很多 n8n 工作流失败，不是节点不会用，而是字段一开始就乱。

例如同一个字段，有时叫：

```text
title
topic
video_name
选题
标题
```

后面就会非常难维护。

### 14.1 推荐字段

```markdown
| 字段 | 说明 | 示例 |
| --- | --- | --- |
| project_id | 项目编号 | 20260710_ai_001 |
| topic | 选题 | AI 视频工作流 |
| audience | 目标观众 | 零基础创作者 |
| platform | 平台 | 抖音 |
| ratio | 画幅 | 9:16 |
| duration | 时长 | 30s |
| status | 状态 | draft / review / approved |
| owner | 负责人 | mk |
| deadline | 截止时间 | 2026-07-15 |
```

### 14.2 状态流转

```text
idea
-> brief_draft
-> brief_review
-> script_draft
-> script_review
-> shot_task
-> generation
-> edit
-> final_review
-> delivered
```

自动化一定要有状态。没有状态，你不知道一个项目到底停在哪一步。

## 15. 人工审核队列

最稳的自动化不是让机器替你决定，而是把需要你看的东西排成队列。

### 15.1 审核队列字段

```markdown
| 字段 | 说明 |
| --- | --- |
| review_id | 审核编号 |
| project_id | 项目编号 |
| review_type | Brief、脚本、分镜、成片等 |
| file_link | 文件位置 |
| risk_note | 自动化发现的风险 |
| decision | approve / revise / reject |
| reviewer | 审核人 |
| decision_time | 审核时间 |
```

### 15.2 审核结果

```text
approve：进入下一步
revise：回到上一步修改
reject：停止当前项目或人工介入
```

这比让工作流一路跑到底更安全。

## 16. Webhook 和平台 API

很多自动化会用到 Webhook 或 HTTP Request。

你可以把它理解成：

```text
某个外部系统通知 n8n：有新任务了
或者 n8n 主动请求某个服务：帮我做这件事
```

### 16.1 适合用 Webhook 的情况

- 表单提交后触发。
- 外部平台生成完成后通知。
- 你在本地脚本里发送一个任务。
- 某个工具回传结果。

### 16.2 HTTP Request 的基本检查

```markdown
| 项目 | 检查 |
| --- | --- |
| URL | 是否是官方或可信接口 |
| Method | GET、POST 等是否正确 |
| Auth | 凭证是否放在安全位置 |
| Body | 请求数据是否包含敏感信息 |
| Response | 返回结果是否记录 |
| Error | 失败时如何处理 |
```

不要在教程、截图或公开仓库里展示真实 Token。

## 17. 云端和自托管怎么选

n8n 可以有云端和自托管等部署方式。新手不要被部署问题拖住。

| 方式 | 适合 | 注意 |
| --- | --- | --- |
| 云端 | 快速开始、少维护 | 账号、数据、费用和权限按官方规则确认 |
| 自托管 | 需要更强控制和私有环境 | 需要服务器、安全、备份、升级能力 |
| 本地测试 | 学习和小样本试验 | 不适合长期稳定运行 |

如果你完全不懂服务器，先从低风险测试工作流开始。不要一上来把客户项目、真实凭证和复杂部署混在一起。

## 18. 版本变更管理

自动化工作流也要有版本。

```text
workflow_ai_video_v01_topic_to_brief
workflow_ai_video_v02_add_script
workflow_ai_video_v03_add_review_queue
```

### 18.1 工作流变更记录

```markdown
| 版本 | 修改 | 原因 | 是否测试 |
| --- | --- | --- | --- |
| v01 | 选题生成 Brief | 建立最小流程 | 是 |
| v02 | 增加脚本草稿 | 扩展流程 | 是 |
| v03 | 增加审核队列 | 防止自动进入后续步骤 | 是 |
```

不要直接在生产工作流上乱改。先复制一份测试，跑小样本，再替换。

## 19. 最小可用工作流

新手第一条工作流建议非常小：

```text
手动触发
-> 输入 topic
-> 生成项目编号
-> 创建 brief 草稿
-> 写入 Markdown
-> 通知你审核
```

这条工作流只有一个 AI 节点，但能让你练会：

- 输入字段。
- 输出格式。
- 文件命名。
- 通知。
- 审核。

掌握这条后，再加脚本、分镜、任务卡。不要第一天就做全链路。

### 19.1 最小输出示例

```markdown
# Brief Draft

## project_id

20260710_ai_001

## topic

AI 视频从提示词进入项目化工作流。

## audience

零基础创作者。

## goal

让观众理解为什么要用项目文件管理创作。

## needs_review

- 是否需要提到具体工具名。
- 是否面向抖音还是 B 站。
- 是否需要商业案例。
```

needs_review 是关键字段。它提醒你：草稿只是草稿。

## 20. 日志和审计

自动化一旦参与商业项目，就必须能回看过程。

你至少要记录：

```markdown
| 项目 | 说明 |
| --- | --- |
| workflow_name | 哪条工作流运行 |
| execution_id | 哪一次执行 |
| trigger_time | 什么时候触发 |
| input_summary | 输入是什么 |
| output_file | 输出到哪里 |
| status | 成功、失败、人工处理中 |
| reviewer | 谁确认 |
| risk_note | 有什么风险 |
```

### 20.1 为什么要记录

日志不是为了显得专业，而是为了回答这些问题：

- 哪个版本的脚本进入了分镜。
- 哪个生成任务用了哪条 Prompt。
- 为什么某个镜头被选中。
- 哪个节点导致成本增加。
- 客户提出修改后改了哪些文件。

没有日志，自动化越多，项目越难解释。

## 21. 模板库复用

n8n 最适合和模板库配合。

你可以把项目模板放在：

```text
templates/
├── brief_template.md
├── script_template.md
├── storyboard_template.md
├── prompt_task_template.md
├── review_template.md
└── delivery_checklist_template.md
```

工作流做的事情不是凭空创造所有内容，而是：

```text
读取模板
-> 填入项目字段
-> 生成草稿
-> 提醒你检查
```

### 21.1 模板变量

```text
{{project_id}}
{{topic}}
{{audience}}
{{platform}}
{{duration}}
{{ratio}}
{{deadline}}
```

模板化的好处是，项目越多，你越不容易漏步骤。

### 21.2 不要模板化的东西

不要把客户判断、最终审美和合规判断做成固定答案。模板只能保证结构，不能保证内容正确。

结构稳定，判断清醒，自动化才会真正帮你省时间。

否则它只会把混乱放大。

先稳，再快。

这就是自动化的顺序。

也是安全底线。

别跳步。稳。

## 22. 常见错误

### 错误 1：一开始就做复杂工作流

修正：先自动创建项目目录和模板文件。

### 错误 2：没有人工确认点

修正：在 Brief、脚本、分镜、导出前设置确认。

### 错误 3：凭证管理混乱

修正：所有凭证放在工具凭证系统，不写进项目文件。

### 错误 4：失败不记录

修正：建立失败表和重试上限。

### 错误 5：自动化没有成本上限

修正：每个项目写最大生成次数和预算。

## 23. 故障排查

### 工作流突然失败

处理：

- 查看 execution 记录。
- 找到失败节点。
- 检查输入是否为空。
- 检查凭证是否过期。
- 手动重跑一个小样本。

### 输出文件命名混乱

处理：

- 统一项目编号。
- 在文件名里加入镜头编号和版本。
- 不允许下载节点使用默认文件名。

### AI 生成结果偏离主题

处理：

- 回到 Brief。
- 缩小输入。
- 在输出里加入“需要人工确认的问题”。
- 不让自动化结果直接进入下一步。

### 成本突然升高

处理：

- 检查是否重复触发。
- 检查是否有无限重试。
- 设置单项目上限。
- 对高成本节点加人工确认。

## 25. 本卷总结

n8n 或类似工具能帮你把重复流程串起来：

```text
收集选题
-> 创建项目
-> 生成草稿
-> 拆任务
-> 归档文件
-> 记录失败
-> 提醒人工确认
```

但自动化不是替代创作判断。越是靠近客户、合规、授权、发布和交付，越需要人看。

最稳的自动化不是“全自动”，而是：

```text
机器做重复动作
人做关键判断
日志记录过程
失败可以回滚
成本可以控制
```
