# LibTV 与 Codex 配置说明

本文件记录配置思路，不保存任何真实密钥、Cookie、Token 或账号信息。

## 1. 配置目标

让 Codex 能够围绕本地项目文件工作，并在需要时把提示词交给 LibTV 或相关 Agent/Skill 执行生成任务。

## 2. 不同连接方式

### 方式 A：手动复制

最稳定，也最适合新手。

流程：

```text
Codex 生成 prompts/shot_01.md
-> 手动复制提示词到 LibTV
-> 下载结果到 outputs/raw
-> 把结果问题告诉 Codex
```

优点：

- 不需要 API 配置。
- 出错少。
- 容易理解平台能力。

缺点：

- 批量效率较低。

### 方式 B：LibTV Skill / Agent

如果使用支持 Codex/Agent 的 LibTV 技能，可以让 Agent 执行创建会话、提交任务、查询状态、下载结果等操作。

注意：

- 以实际插件或 skill 文档为准。
- 不要把密钥写进公开仓库。
- 生成成本需要单独记录。
- 出错时先回到手动复制方式验证提示词是否可用。

### 方式 C：API / MCP

适合批量生产或团队流水线。

建议只有在下面条件满足时再使用：

- 已经熟悉手动生成流程。
- 已经有稳定提示词模板。
- 知道每次生成的大致成本。
- 有清晰的失败重试策略。

## 3. 本地配置文件建议

可以创建本地 `.env`，但不要提交：

```env
LIBTV_API_KEY=
LIBTV_OUTPUT_DIR=outputs/raw
DEFAULT_VIDEO_RATIO=9:16
DEFAULT_VIDEO_DURATION=5
```

`.gitignore` 应加入：

```text
.env
*.token
*.cookie
```

## 4. Codex 操作边界

Codex 可以做：

- 生成提示词。
- 整理项目目录。
- 维护迭代记录。
- 调用本地命令。
- 根据日志修正文件。

Codex 不应擅自做：

- 消耗付费额度的大批量生成。
- 公开提交密钥。
- 在未确认成本时循环重试。
- 把推断写成事实。
