# AI 视频制作百科完整版扩写设计

## 1. 背景

当前仓库已经完成项目骨架、18 卷目录、Ross 抖音案例核验分析、Codex + LibTV 工作流初稿和若干模板资源。但它还不是 20 万字级完整教程。当前内容更接近“目录 + 样章 + 第一个案例包”。

用户明确目标是：在教程补成完整版之后再开始系统学习。因此后续工作要优先服务“可自学、可查阅、可实操”，而不是只追求字数。

## 2. 总目标

把 `AI-Film-Book` 扩写为一套长期维护的 Markdown 电子书：

- 目标总量：20 万字以上。
- 结构：18 卷主教程 + Prompt 库 + 运镜模板 + 分镜模板 + 风格模板 + API 配置 + 故障排查 + 参数速查 + 工具排行榜。
- 读者：零基础 AI 视频学习者、短视频创作者、商业视频交付者。
- 输出形态：GitHub 仓库中的 Markdown 文档，后续可导出 PDF / EPUB。
- 写作重点：原理、工作流、实战、复盘和可复制模板。

## 3. 第一阶段范围

用户批准采用方案 A，并要求先扩写第 01-05 卷。

第一阶段文件：

1. `第01卷_AI视频基础.md`
2. `第02卷_环境搭建.md`
3. `第03卷_ChatGPT与Codex.md`
4. `第04卷_ContextEngineering.md`
5. `第05卷_LibTV完全教程.md`

第一阶段目标不是写完整 20 万字，而是先完成百科第一部分：从零基础到能理解并复刻 Codex + LibTV 工作流。

建议第一阶段总量：

- 第 01 卷：约 12,000 到 18,000 字。
- 第 02 卷：约 10,000 到 15,000 字。
- 第 03 卷：约 12,000 到 18,000 字。
- 第 04 卷：约 12,000 到 18,000 字。
- 第 05 卷：约 15,000 到 25,000 字。

第一阶段合计约 61,000 到 94,000 字，足够作为完整学习入口。

## 4. 第一阶段学习结果

第 01-05 卷完成后，读者应能做到：

- 解释 AI 视频制作不是“一句提示词出片”，而是脚本、分镜、画面、声音、剪辑和复盘的组合工程。
- 搭建一个长期维护的本地 AI 视频项目目录。
- 用 Git 记录教程、提示词和作品迭代。
- 理解 ChatGPT 与 Codex 在创作中的分工。
- 理解 Context Engineering 为什么比单条提示词更重要。
- 用 Codex 生成 brief、脚本、分镜、风格手册、角色圣经和镜头提示词。
- 把提示词投喂给 LibTV 或同类工具生成素材。
- 记录失败样本，并让 Codex 参与下一轮提示词修正。
- 复刻 Ross 案例中的“AI 虚拟口播 + 电影感背景 + 黄色字幕 + 多模型组合”工作流。

## 5. 每卷固定写作结构

每卷使用统一结构，但允许根据主题调整章节数量：

1. 学习目标
2. 本卷导读
3. 核心概念
4. 底层机制
5. 标准工作流
6. 实战案例
7. Prompt / 模板
8. 常见错误
9. 故障排查
10. 参数或清单速查
11. 本卷总结
12. 课后练习
13. 延伸阅读与核验记录

每一卷都必须回答三个问题：

- 这个主题解决 AI 视频制作的哪一环？
- 输入是什么，输出是什么？
- 失败时应该如何定位问题？

## 6. 分卷设计

### 第 01 卷：AI 视频基础

目标：建立学习者的全局认知。

扩写重点：

- AI 视频的四层结构：创意层、叙事层、镜头层、生产层。
- AI 视频和传统视频制作的关系。
- 文生视频、图生视频、视频生视频、数字人、口播视频的差别。
- 为什么 AI 视频没有电影感。
- 为什么人物一致性、空间连续性、剪辑节奏比单条提示词更重要。
- 15 秒试验片的最小闭环。

### 第 02 卷：环境搭建

目标：让读者建立可持续工作的本地项目环境。

扩写重点：

- 项目目录规范。
- Git 与版本管理。
- 素材命名规范。
- `.env`、密钥和账号安全。
- 本地作品项目模板。
- Codex 工作方式。
- LibTV 或同类工具接入前准备。
- 剪辑软件和导出目录准备。

### 第 03 卷：ChatGPT 与 Codex

目标：让读者理解“聊天助手”和“项目 Agent”的区别。

扩写重点：

- ChatGPT 适合创意、解释、改写、讨论。
- Codex 适合读文件、改文件、批量生成、维护上下文。
- Codex 在 AI 视频中的 12 个典型任务。
- 如何给 Codex 下任务。
- 如何让 Codex 先读项目文件再生成。
- 如何避免 Codex 编造工具能力。
- 如何使用 Codex 做分镜、提示词、复盘和 Git 提交。

### 第 04 卷：Context Engineering

目标：把“手搓提示词”升级为“项目上下文工程”。

扩写重点：

- Prompt、Context、Workflow 的区别。
- Brief、Script、Storyboard、Style Guide、Character Bible、Iteration Log 的关系。
- 上下文污染与上下文丢失。
- 如何分层组织信息。
- 如何让不同模型共享同一套项目约束。
- Ross 案例中的上下文结构复原。
- 可复制上下文模板。

### 第 05 卷：LibTV 完全教程

目标：讲清楚 LibTV 在 AI 视频生产中的位置和用法。

扩写重点：

- LibTV 的工作流定位。
- 手动复制提示词、Skill/Agent、API/MCP 三种使用方式。
- 文生图、图生视频、文生视频、虚拟口播素材的差异。
- 如何把 Codex 输出转成 LibTV 任务。
- 如何做单镜头生成。
- 如何做多版本筛选。
- 如何记录成本和失败样本。
- 如何复刻 Ross 案例的虚拟口播视频。

涉及平台实时信息时，应优先标注“需以实际平台为准”。价格、模型名、按钮位置和账户权限属于高变动信息，不写成永久事实。

## 7. 资料核验规则

第一阶段会涉及 OpenAI/Codex、LibTV、多模型视频生成工具。规则如下：

- OpenAI/Codex 相关事实以官方 OpenAI 文档和官方 GitHub 仓库为准。
- LibTV 相关事实以实际页面、公开仓库、公开说明或用户可提供截图为准。
- 第三方工具功能、价格、模型名称可能变化，必须写明核验日期或“以平台实际为准”。
- 未核验信息可以作为“推断”或“建议工作流”出现，但不能写成事实。
- 案例拆解必须区分页面信息、画面观察、制作推断。

## 8. 写作质量标准

每个章节应满足：

- 新手能读懂，不要求先懂导演、剪辑或编程。
- 创作者能拿去复用，不停留在概念解释。
- 每个核心概念至少给一个例子。
- 每个工作流至少给一条可复制指令或模板。
- 每个工具章节都说明输入、输出、失败点。
- 避免堆砌官网介绍。
- 避免空泛鼓励语。
- 避免把 AI 说成万能自动化。

## 9. 交付节奏

第一阶段建议拆为多个提交：

1. 扩写第 01 卷。
2. 扩写第 02 卷。
3. 扩写第 03 卷。
4. 扩写第 04 卷。
5. 扩写第 05 卷。
6. 补第一阶段学习路线、14 天训练营和阶段练习。
7. 全局术语统一、链接检查、提交并推送。

每次提交前做以下检查：

- 文件是否在目标卷范围内。
- 是否有空章节、临时文本或尚未补完的段落。
- 是否有过时或未核验的事实表述。
- 是否误提交本地临时素材。
- Git 工作区是否干净。

## 10. 非目标

第一阶段不做：

- 不完成第 06-18 卷正文。
- 不导出 PDF / EPUB。
- 不提交视频素材、截图或大型媒体文件。
- 不写不可核验的价格排行榜。
- 不把任一工具描述成唯一正确选择。

这些内容留到后续阶段。

## 11. 验收标准

第一阶段完成时应满足：

- 第 01-05 卷从骨架升级为完整可读教程。
- 每卷都有学习目标、原理、工作流、案例、模板、错误、练习。
- `学习路线.md` 更新为“先学第 01-05 卷”的明确路径。
- 新增或更新一个 14 天入门训练营。
- Ross 案例能作为贯穿第 01-05 卷的示例。
- GitHub 远程仓库同步到最新提交。
