# 第 05 卷：LibTV 完全教程

## 学习目标

本卷讲 LibTV 在 AI 视频工作流中的位置。读完之后，你应该能做到：

- 知道 LibTV 或同类平台解决的是“生成素材”和“组织生成任务”的问题。
- 知道 LibTV 不替你完成选题、导演判断、剪辑节奏和商业决策。
- 理解手动复制、Skill/Agent、API/MCP 三种使用方式的差别。
- 把 Codex 生成的分镜和提示词转成可执行的生成任务。
- 按单镜头生成、多版本筛选、失败记录、成本记录的方式复盘。
- 设计一个 Ross 风格虚拟口播视频的复刻流程。

本卷不会编造具体价格、按钮路径和隐藏参数。平台 UI、模型名称、权限和计费规则可能随时间变化，实际操作时以 LibTV 或对应平台当前页面为准。

## 本卷导读：LibTV 在工作流里的位置

前四卷已经建立了基础：

```text
第 01 卷：知道 AI 视频是什么
第 02 卷：建立项目目录
第 03 卷：学会指挥 Codex
第 04 卷：建立上下文工程
```

第 05 卷进入生成环节。

但要先说清楚：LibTV 不是魔法按钮。它更像一个生成平台或生成工作流入口。你把 Codex 整理出来的项目上下文、分镜和提示词交给它，它帮你生成图片或视频素材。素材生成后，仍然需要筛选、复盘和剪辑。

推荐理解：

```text
Codex：整理项目、写脚本、拆分镜、写提示词、复盘失败
LibTV：根据提示词和素材生成图片/视频结果
剪辑软件：筛选、组接、字幕、声音、导出
人类：判断方向、选择结果、控制成本、决定最终表达
```

如果你跳过 Codex 和上下文工程，直接把一句话丢给 LibTV，也可能生成好看的东西。但那更像抽奖，不像生产。

## 1. LibTV 解决什么问题

LibTV 在本书里的定位是：AI 视频素材生成和 Agent 化创作入口。

根据公开的 `libtv-labs/libtv-skills` 仓库说明，LibTV Skill 面向 AI Agent，通过 LibLib.tv OpenAPI 提供 AI 生图、AI 生视频等 AIGC 能力，Agent 可以创建会话、发送创作指令、查询生成进展，并获取图片或视频结果。这个能力说明了一个方向：视频生成平台正在从“人手动点按钮”走向“Agent 可调用的创作流程”。

对新手来说，可以先记住三个作用：

### 1.1 生成素材

把文字提示词、参考图或项目描述变成图片/视频素材。

### 1.2 组织任务

把多个镜头、多轮生成、多版本结果放到一个工作流里管理。

### 1.3 连接 Agent

通过 Skill、API 或 MCP 等方式，让 Codex 或其他 Agent 更自然地参与创作流程。

这也是 Ross 案例标题里“Codex + LibTV”组合有意义的原因：Codex 组织上下文，LibTV 承接生成任务。

## 2. LibTV 不解决什么问题

学习工具时，知道它不解决什么同样重要。

LibTV 不负责：

- 替你判断视频主题是否有价值。
- 替你决定目标观众。
- 替你保证分镜合理。
- 替你保证每个镜头都服务叙事。
- 替你完成最终剪辑节奏。
- 替你判断商业授权和客户需求。
- 替你永久保证人物一致性。

如果你的 brief 很模糊，LibTV 不会自动变清晰。如果你的分镜没有目的，生成素材也可能没有叙事价值。如果你的角色圣经不存在，人物漂移的概率就会提高。

正确心态是：

```text
LibTV 是生成环节的强工具，不是创作决策的替身。
```

## 3. 手动复制、Skill/Agent、API/MCP 三种使用方式

### 3.1 手动复制

这是最推荐的新手方式。

流程：

```text
Codex 生成 prompts/shot_01.md
-> 人工复制提示词到 LibTV
-> 平台生成视频
-> 下载结果到 outputs/raw/
-> 人工筛选到 outputs/selected/
-> 把失败原因写入 iteration_log.md
-> 让 Codex 修改对应提示词
```

优点：

- 不需要配置 API。
- 不容易误消耗大量额度。
- 新手能看懂每一步。
- 容易判断问题来自提示词还是平台。

缺点：

- 批量效率低。
- 需要手动下载和命名。

第一阶段建议先用手动复制。因为你还在学习输入和输出，不要急着自动化。

### 3.2 Skill/Agent

Skill/Agent 方式指让 Agent 直接调用 LibTV 的能力，例如创建会话、提交任务、查询状态、获取结果。

优点：

- 更接近 Ross 标题里的“Agent 工作流”。
- 可以减少手动复制。
- 适合多镜头、多轮迭代。

风险：

- 可能消耗额度更快。
- 如果提示词不清晰，Agent 会批量放大错误。
- 具体能力取决于当前 Skill、平台权限和账号配置。

新手应该先理解手动流程，再尝试 Skill。

### 3.3 API/MCP

API/MCP 方式适合更高级的自动化。

适合场景：

- 批量生成。
- 团队流水线。
- 自动下载结果。
- 自动写入项目目录。
- 和 n8n、数据库、内容系统连接。

不适合新手一开始使用，因为你还不知道什么是好输入、什么是坏输出。自动化会让错误变得更快。

## 4. 文生图、图生视频、文生视频和虚拟口播素材

LibTV 或同类平台可能支持多种生成方式。具体名称以平台当前页面为准，但概念可以先这样理解。

| 类型 | 输入 | 输出 | 适合场景 | 风险 |
| --- | --- | --- | --- | --- |
| 文生图 | 文本 | 图片 | 概念图、角色图、参考图 | 需要再转视频 |
| 图生视频 | 图片 + 动作提示 | 视频 | 让角色、产品、场景动起来 | 动作大时变形 |
| 文生视频 | 文本 | 视频 | 概念镜头、环境镜头 | 可控性相对低 |
| 虚拟口播素材 | 角色/音频/脚本 | 口播视频 | 教程、讲解、知识短视频 | 嘴型、表情、手势不稳定 |

Ross 案例更像“虚拟口播 + 背景切换 + 字幕后期 + 多模型组合”的作品。

## 5. 从 Codex 输出到 LibTV 任务

Codex 输出不应该是一大段混合提示词，而应该是单镜头任务。

### 5.1 输入文件

生成前，Codex 至少应该读取：

```text
brief.md
storyboard.md
style_guide.md
prompts/shot_01.md
```

如果有固定人物，还要读取：

```text
character_bible.md
```

### 5.2 单镜头任务格式

```markdown
# Shot 01

## 镜头目的

## 输入类型

文生视频 / 图生视频 / 口播素材

## 中文提示词

## English Prompt

## Negative Prompt

## 参数建议

## 生成前检查

## 可能失败点
```

这个结构可以直接复制到 `prompts/shot_01.md`。

### 5.3 不要把字幕交给模型

如果画面里需要出现清晰中文，建议后期添加。视频模型生成可读文字通常不稳定。

提示词中可以写：

```text
abstract interface, no readable text
```

然后在剪辑软件中添加真实字幕。

## 6. 单镜头生成流程

单镜头生成是新手最应该掌握的流程。

### 6.1 生成前

检查：

| 项目 | 问题 |
| --- | --- |
| 镜头目的 | 这个镜头为什么存在 |
| 主体 | 谁或什么在画面里 |
| 动作 | 主体做什么 |
| 场景 | 发生在哪里 |
| 景别 | 远景、中景、近景、特写 |
| 运镜 | 固定、慢推、横移等 |
| 光线 | 主光、轮廓光、屏幕光等 |
| 风格 | 是否引用 style guide |
| 负面约束 | 避免什么 |
| 输出目录 | 下载后放哪里 |

### 6.2 生成中

不要一次性生成太多镜头。建议：

```text
先选 S01
生成 2 个版本
下载并命名
观察失败点
再决定是否生成 S02
```

### 6.3 生成后

下载到：

```text
outputs/raw/
```

命名：

```text
S01_V01_raw_workspace_opening.mp4
S01_V02_raw_workspace_opening.mp4
```

然后筛选：

```text
outputs/selected/
outputs/rejected/
```

## 7. 多版本筛选

不要只看“好不好看”，要按镜头目的筛选。

### 7.1 筛选表

```markdown
| 镜头 | 文件 | 画面质量 | 是否符合目的 | 风格一致 | 主要问题 | 结论 |
| --- | --- | --- | --- | --- | --- | --- |
| S01 | S01_V01_raw.mp4 | 中 | 是 | 是 | 屏幕文字乱码 | 部分可用 |
| S01 | S01_V02_raw.mp4 | 高 | 否 | 否 | 场景变成街道 | 不可用 |
```

### 7.2 筛选标准

优先级：

1. 是否服务镜头目的。
2. 是否保持风格一致。
3. 是否有严重技术问题。
4. 是否适合剪辑。
5. 是否足够好看。

注意顺序。好看但不服务目的的镜头，不是好素材。

## 8. 失败样本记录

失败样本要写进 `iteration_log.md`。

模板：

```markdown
## 第一轮生成

| 镜头 | 文件 | 问题 | 归因 | 是否可用 | 下一步 |
| --- | --- | --- | --- | --- | --- |
```

### 8.1 常见归因

| 归因 | 说明 | 下一步 |
| --- | --- | --- |
| 需求问题 | 镜头目的不清 | 改 storyboard |
| 提示词问题 | 描述不够具体 | 改 prompt |
| 素材问题 | 参考图不稳定 | 换参考图 |
| 模型问题 | 当前模型不擅长 | 换模型或生成方式 |
| 后期问题 | 素材可用但需要剪辑处理 | 留到剪辑阶段 |

### 8.2 给 Codex 的复盘指令

```text
请读取 iteration_log.md 和 prompts/。
只修改标记为“否”或“部分可用”的镜头提示词。
不要修改已经可用的镜头。
每个修改都要说明原因。
```

## 9. 成本记录

AI 视频制作需要成本意识。成本不只包括钱，还包括时间和失败次数。

### 9.1 cost_log.md 模板

```markdown
# Cost Log

| 日期 | 工具 | 镜头 | 任务 | 次数 | 单价 | 总价 | 结果 | 备注 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| 2026-07-10 | LibTV | S01 | 文生视频 | 2 | 以平台为准 | 以平台为准 | 1 个部分可用 | 屏幕文字乱码 |
```

### 9.2 成本控制原则

- 第一轮每个镜头只生成 2 个版本。
- 先生成关键镜头，不要全片同时生成。
- 对失败镜头先改提示词，再重生成。
- 记录每次失败原因。
- 不要为了一个不重要镜头无限重试。

## 10. 人物一致性处理

虚拟口播、短剧、人物广告都绕不开人物一致性。

### 10.1 生成前

准备：

```text
character_bible.md
参考图
固定服装
固定发型
固定姿态范围
```

### 10.2 提示词中要固定什么

示例：

```text
same young Asian male presenter, short black hair, gray-green hoodie, calm focused expression, facing camera, documentary talking-head style
```

但不要只写 `same character`。模型需要具体特征。

### 10.3 降低漂移的方法

- 单镜头不要太长。
- 不要频繁改变机位。
- 不要同时做复杂手势。
- 尽量使用稳定参考图。
- 每轮生成后对比脸、服装、发型。

## 11. 背景切换和风格统一

Ross 案例中，背景会从城市夜景切到火箭发射场，但人物和字幕风格保持相对统一。

要做到这一点，需要区分：

```text
固定元素：人物、字幕、口播风格、色彩质感
可变化元素：背景、镜头景别、画面示例
禁止漂移项：人物年龄、服装、整体风格
```

背景切换不是随机换图，而是服务叙事：

| 背景 | 作用 |
| --- | --- |
| 城市夜景 | 建立科技感和电影感 |
| 工作室 | 建立创作场景 |
| 火箭发射场 | 表达成本、技术、速度或宏大感 |
| 手机播放画面 | 回到发布结果 |

## 12. Ross 案例复刻流程

这里不是复刻作者的原始文件，而是复刻工作流。

### 12.1 项目目标

```text
做一条 60 到 75 秒视频，表达 Codex + LibTV 工作流正在替代手搓提示词。
```

### 12.2 镜头结构

| 镜头 | 内容 | 目的 |
| --- | --- | --- |
| S01 | 高质量 AI 人物或场景开场 | 证明画面质量 |
| S02 | 虚拟口播人物出现 | 建立讲述者 |
| S03 | 展示成本对比 | 降低门槛感 |
| S04 | 提到多模型组合 | 提供方法论 |
| S05 | 回到 Codex + LibTV 工作流 | 强化标题观点 |
| S06 | 结语和发布引导 | 留下记忆点 |

### 12.3 Codex 指令

```text
请根据 Ross 案例拆解，创建一个 60 秒 AI 视频项目。
主题是：全程用 Codex 和 LibTV 完成，手搓提示词时代过去。
请先生成 brief.md、script.md、storyboard.md、style_guide.md、character_bible.md。
不要生成未核验的平台价格和具体按钮路径。
```

### 12.4 LibTV 生成策略

- 先生成虚拟口播人物测试镜头。
- 再生成背景切换镜头。
- 字幕全部留给剪辑软件。
- 每个镜头保留 2 到 4 个版本。
- 成本和失败都记录。

### 12.5 后期策略

- 黄色大字幕强化短视频阅读。
- 用剪辑节奏压缩等待感。
- 用背景切换增加视觉变化。
- 保留“成本低”“多模型”“工作流替代提示词”三个传播点。

## 13. 完整操作案例：30 秒工作流演示片

下面用一个 30 秒项目演示从 Codex 到 LibTV 的完整流程。这个案例比 15 秒练习更完整，但仍然控制在新手能理解的范围内。

### 13.1 项目目标

```text
做一条 30 秒短视频，展示“AI 视频不是一句提示词，而是一套 Codex + LibTV 工作流”。
```

目标观众：

```text
刚接触 AI 视频，看到别人作品很震撼，但不知道从哪里开始的人。
```

核心观点：

```text
先用 Codex 把项目拆清楚，再用 LibTV 或同类工具生成素材，最后剪辑复盘。
```

### 13.2 30 秒镜头表

| 镜头 | 时长 | 画面 | 生成方式 | 目的 |
| --- | --- | --- | --- | --- |
| S01 | 4s | 高质量 AI 画面开场，城市夜景人物近景 | 文生视频或图生视频 | 证明画面吸引力 |
| S02 | 5s | 桌面上打开项目文件夹和 brief | 文生视频 | 展示项目化起点 |
| S03 | 5s | 分镜表、风格手册、提示词文件快速切换 | 文生视频或后期录屏 | 展示 Codex 产物 |
| S04 | 5s | LibTV 生成素材界面或抽象生成过程 | 文生视频/界面后期 | 展示生成环节 |
| S05 | 5s | 多版本素材对比，选中一个可用镜头 | 后期剪辑 | 展示筛选 |
| S06 | 4s | 剪辑时间线组接、加字幕 | 后期素材 | 展示剪辑 |
| S07 | 2s | 手机播放成片，出现标题 | 文生视频或实拍 | 完成闭环 |

### 13.3 先让 Codex 创建文件

指令：

```text
你是我的 AI 视频项目制片助理。
我要做一条 30 秒视频，主题是“AI 视频不是一句提示词，而是一套 Codex + LibTV 工作流”。
请创建 brief.md、script.md、storyboard.md、style_guide.md、iteration_log.md、cost_log.md。

要求：
1. 面向 AI 视频新手。
2. 平台是抖音，画幅 9:16。
3. 风格是真实纪录片 + 轻微未来感。
4. 不要写具体平台价格。
5. 字幕由后期添加，不要求视频模型生成可读中文。
```

### 13.4 再生成单镜头任务卡

每个镜头都应该变成任务卡。任务卡比普通提示词更完整。

```markdown
# S01 生成任务卡

## 镜头目的

用强画面抓住观众，证明 AI 视频可以有质感。

## 输入类型

文生视频，必要时先文生图再图生视频。

## 主体

一位年轻创作者或虚拟人物，城市夜景背景。

## 场景

雨后城市天台或街道，远处高楼灯光虚化。

## 动作

人物看向镜头，轻微呼吸和表情变化，不做复杂手势。

## 镜头

中近景，固定机位或极慢推近。

## 光线

冷色城市环境光，暖色侧后方轮廓光。

## 中文提示词

夜晚雨后的城市天台，远处高楼灯光虚化，一位年轻创作者面对镜头，表情自然克制，轻微呼吸和眼神变化。中近景，固定机位，冷色城市环境光和暖色侧后方轮廓光，真实电影摄影，低饱和，浅景深，轻微胶片颗粒。

## Negative Prompt

cartoon, anime, distorted face, extra fingers, unreadable text, logo, watermark, over sharpened, plastic skin

## 生成参数建议

9:16，4 秒，慢节奏，动作克制。

## 可能失败点

人物脸部不稳定，背景过度科幻，画面出现乱码文字。
```

任务卡的好处是：它不仅告诉平台生成什么，也告诉你为什么生成、如何判断失败。

### 13.5 生成批次安排

不要一次性生成 7 个镜头。建议批次：

| 批次 | 镜头 | 目的 |
| --- | --- | --- |
| 第 1 批 | S01、S02 | 验证整体风格 |
| 第 2 批 | S03、S04 | 验证工作流表达 |
| 第 3 批 | S05、S06 | 验证后期素材 |
| 第 4 批 | S07 | 验证结尾 |

每批只生成 2 个版本，复盘后再继续。

### 13.6 第一轮记录示例

```markdown
## 第一轮生成记录

| 镜头 | 文件 | 结果 | 问题 | 归因 | 下一步 |
| --- | --- | --- | --- | --- | --- |
| S01 | S01_V01_raw_city_presenter.mp4 | 部分可用 | 人物自然，但背景太科幻 | 风格漂移 | 降低赛博朋克感，强调真实城市 |
| S01 | S01_V02_raw_city_presenter.mp4 | 不可用 | 人物脸部变形 | 模型/动作问题 | 减少表情变化，缩短动作描述 |
| S02 | S02_V01_raw_workspace_folder.mp4 | 部分可用 | 桌面好看，屏幕文字乱码 | 文字问题 | 改成抽象界面，后期加字 |
| S02 | S02_V02_raw_workspace_folder.mp4 | 可用 | 风格统一，可剪辑 | 无 | 移动到 selected |
```

### 13.7 让 Codex 修改下一轮


## 14. 质量评分表

生成素材时，可以给每个版本打分。评分不是为了精确，而是为了帮助选择。

| 维度 | 1 分 | 3 分 | 5 分 |
| --- | --- | --- | --- |
| 镜头目的 | 完全不符合 | 部分符合 | 完全服务该镜头 |
| 风格一致 | 和项目风格冲突 | 有少量漂移 | 与 style guide 一致 |
| 主体稳定 | 主体变形严重 | 有小问题 | 主体稳定清楚 |
| 动作自然 | 动作混乱 | 基本可用 | 自然可信 |
| 剪辑可用 | 很难剪 | 可勉强使用 | 容易进入时间线 |
| 技术问题 | 明显乱码/变形 | 有小瑕疵 | 无明显问题 |

评分示例：

```markdown
| 镜头 | 文件 | 目的 | 风格 | 主体 | 动作 | 剪辑 | 技术 | 总分 | 结论 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |
| S01 | V01 | 5 | 4 | 3 | 4 | 4 | 3 | 23 | 部分可用 |
| S01 | V02 | 2 | 2 | 1 | 2 | 2 | 1 | 10 | 不可用 |
```

一般来说，低于 18 分的素材不建议进入剪辑。18 到 24 分可以作为部分素材。25 分以上才适合作为主镜头。

## 16. 从手动流程升级到 Agent 流程

新手第一阶段用手动复制。等你能稳定完成以下动作，再考虑升级：

```text
能写清楚 brief
能写清楚 storyboard
能判断每个镜头失败原因
能记录 cost_log
能控制每轮生成次数
能筛选 selected/rejected
```

升级后，Agent 可以帮你：

- 批量读取 prompts。
- 提交生成任务。
- 查询任务状态。
- 下载结果。
- 把结果放进指定目录。
- 生成初步复盘表。

但你仍然要人工判断：

- 哪个版本进入剪辑。
- 哪个失败值得重试。
- 哪个镜头应该换方案。
- 成本是否值得继续。

Agent 流程不是让人退出，而是把重复动作交给工具。

## 17. LibTV 卷的学习路径

完全新手读本卷时，可以按这个顺序练：

### 第 1 步：只读懂输入输出

不要急着注册、充值或生成。先写清楚：

```text
输入：prompts/shot_01.md
输出：outputs/raw/S01_V01_raw.mp4

```

### 第 2 步：准备一个镜头

只准备 S01，不要准备全片。

检查：

- 镜头目的。
- 主体。
- 场景。
- 动作。
- 光线。
- 风格。
- 负面约束。

### 第 3 步：生成两个版本

只生成两个版本。下载后命名。

### 第 4 步：写复盘

再生成两个版本，对比第一轮是否变好。

只要这个小循环跑通，你就学会了 LibTV 工作流的核心。

## 18. 常见错误

### 错误 1：一开始就用自动化

新手还没理解输入输出时，自动化只会更快消耗额度。

修正：先手动复制，跑通 2 到 3 个镜头。

### 错误 2：一次生成整条视频

后果：失败后无法定位。

修正：单镜头生成，单镜头复盘。

### 错误 3：把字幕交给视频模型

后果：文字乱码。

修正：生成画面时弱化文字，字幕后期添加。

### 错误 4：不记录成本

后果：不知道哪里浪费。

修正：每轮生成写 `cost_log.md`。

### 错误 5：迷信某个模型

后果：任务不匹配时反复失败。

修正：按镜头任务选择工具，而不是按热度选择工具。

## 19. 故障排查

### prompt too broad

表现：画面元素很多，但没有重点。

处理：回到 storyboard，明确主体、动作和镜头目的。

### model mismatch

表现：当前模型一直做不好某类任务。

处理：换生成方式，例如先文生图再图生视频。

### character drift

表现：人物年龄、脸型、服装变化。

处理：补 character_bible，减少复杂动作，使用参考图。

### background drift

表现：场景从工作室变成街道，或风格突然变化。

处理：强化 style_guide，写清固定元素和禁止漂移项。

### unreadable text

表现：屏幕、招牌、字幕乱码。

处理：提示词中写抽象界面，不要求可读文字，后期加字。

### cost overrun

表现：一个镜头反复生成很多次仍不能用。

处理：停止重试，先归因，再改分镜、提示词或模型。

## 20. 参数速查

生成前检查：

| 项目 | 推荐 |
| --- | --- |
| 画幅 | 抖音优先 9:16 |
| 单镜头时长 | 新手先 3 到 5 秒 |
| 版本数 | 第一轮 2 个 |
| 字幕 | 后期添加 |
| 参考图 | 固定人物时尽量准备 |
| 负面约束 | 卡通、变形、乱码、多余人物 |
| 输出路径 | `outputs/raw/` |
| 复盘路径 | `iteration_log.md` |
| 成本路径 | `cost_log.md` |

## 21. 本卷总结

LibTV 或同类平台是 AI 视频工作流中的生成环节。它很重要，但不是全部。

稳定流程是：

```text
Codex 整理上下文
-> LibTV 生成素材
-> 人类筛选
-> Codex 复盘失败
-> 剪辑软件完成成片
```

第一阶段最重要的不是自动化，而是理解每个输入和输出。先手动跑通，再考虑 Skill、Agent、API 和 MCP。

### 21.1 本卷验收标准

学完本卷后，你应该能在不依赖别人提示的情况下完成一个单镜头生成循环：

```text
从 storyboard 选择一个镜头
-> 写 prompts/shot_01.md
-> 在 LibTV 或同类平台生成 2 个版本
-> 下载到 outputs/raw/
-> 按命名规范改名
-> 选择 selected 或 rejected
-> 写入 iteration_log.md
-> 让 Codex 根据失败记录修改下一轮 prompt
```

如果你能说清楚这条链路里每一步的输入、输出和失败风险，就说明你已经掌握了第一阶段的生成工作流。之后再学习自动化，才不会被工具带着跑。

记住本卷的核心原则：先小后大，先单镜头后整片，先手动理解再自动化。真正稳定的 AI 视频生产，不是一次生成很多素材，而是每一轮都知道为什么失败、下一步改哪里。

如果这个循环还跑不顺，不要急着扩展到整片。先把一个镜头练到能稳定复盘，再把同样的方法复制到第二个镜头。对新手来说，这比追求一次做出完整大片更重要，也更接近真实项目里的工作节奏。
